# ctxloom

**让 AI 开发协作有记忆、有流程、越用越聪明。**

普通 AI 辅助开发有三个根本痛点：每次会话从零开始、决策散落在对话里、经验永远留不下来。ctxloom 用一套 Claude Code 技能插件正面解决这三件事——它在你的项目里维护一份 `.context/` 活文档，把每次开发的需求、设计、踩的坑、定的约定都沉淀下来，下次直接复用。

## 它和其他 AI 编程工具有什么不同

| | 普通 AI 编程工具 | ctxloom |
|---|---|---|
| **项目记忆** | 每次会话从零 | `.context/` 持久化项目背景、架构、规则 |
| **开发流程** | 自由发挥，容易跑偏 | 按任务规模自动路由，关键节点人工确认 |
| **经验沉淀** | 对话结束即消失 | 复盘自动回写规则库与经验库 |
| **语言适配** | 内置特定语言规范 | 规则存在项目里，技能本身语言无关 |
| **评审把关** | 无 | 需求、设计阶段各有独立评审智能体 |

## 核心：`.context/` 知识飞轮

ctxloom 在你的项目根目录维护一套 `.context/` 活文档系统。每次开发的产出——需求决策、技术方案、执行计划、踩的坑、定的约定——都沉淀进来，下一次开发直接复用，不再重来。

```
.context/
├── project.md          # 项目背景、技术栈、整体架构（权威来源）
├── learnings.md        # 踩坑经验与临时约定（复盘后滚动追加）
├── pending.md          # 待确认问题的暂存区
├── rules/              # 编码前的规则基线（语言无关，按项目填写）
│   ├── build.md        # 构建命令（计划/执行/提交阶段强依赖）
│   ├── testing.md      # 测试框架与命令（计划/执行/提交阶段强依赖）
│   ├── security.md     # 本项目安全高危点（设计/提交阶段强依赖）
│   ├── code-style.md   # 命名/格式等外观风格
│   ├── coding.md       # 分层/异常/事务等工程约定
│   └── other.md        # 其他项目特有约定
├── requirements/       # 需求活文档（每功能一份，贯穿全生命周期）
├── designs/            # 技术设计活文档
├── tasks/              # 执行计划活文档
│   └── current.md      # 当前在途功能的轻量索引
└── archive/            # 已交付功能的提炼归档
```

**活文档而非快照。** 同一功能的需求、设计、执行用同名文件贯穿（`coupon.md` 从 `requirements/` 走到 `archive/`），始终反映当前事实，历史靠 git 追溯，不堆日期戳文件。

**规则库与经验库分工。** `rules/` 存放能指导未来任意新代码的通用规则；`learnings.md` 存放偶发教训。每次复盘后，够格的经验提炼进规则，不够格的留在经验库——项目越做规则越准。

**语言无关。** 技能本身不写死任何语言命令，所有项目特有的构建/测试/安全内容固化在 `rules/` 里，技能按需读取，可复用到 Java、Vue、Go 等任何项目。

## 两个入口：dev 与 bugfix

日常开发你几乎只需要记住两个入口，剩下的交给它们自动调度：

- **`dev`** —— 所有功能开发的统一入口。它先评估任务规模，再自动路由到四条路径之一，不为小任务套重流程，也不让大任务裸奔：

  | 路径 | 适用 | 怎么走 |
  |---|---|---|
  | **极简** | 方案唯一、≤10 行、单模块、低风险 | 直接改，改完告知怎么验 |
  | **快速** | 需求清楚、2-3 文件、少量决策 | 先讲方案并确认，再动代码 |
  | **标准** | 中等复杂、跨 2-3 模块、有设计决策 | 调用 `1`→（可选 `2`）→`4`→`5` |
  | **完整** | 需求模糊、跨多模块、架构影响、高风险 | 完整跑 `1`→`2`→`3`→`4`→`5`→`6` |

- **`bugfix`** —— 修复缺陷的并行主线。它的主轴是新功能流水线里没有的两道工序：**用证据定位根因**和**按复杂度分流**。小 bug 复现→根因→修复一气呵成，复杂 bug 才升级复用 `2`/`3`/`4`，不为简单问题套重流程。

## 0–6 流水线（dev 在背后调度，也可单独调用）

```
0-init          → 建立 .context/ 知识地基
1-requirements  → 需求澄清（独立评审把关）
2-design        → 技术设计（独立评审 + 防过度设计）
3-plan          → 拆解实现步骤（含测试策略）
4-execute       → 落地编码（按计划、按步验证）
5-commit        → 质量清单 + 归档 + 提交
6-retrospective → 复盘回写规则库与经验库
```

每个技能都可**单独运行**，不强制走完整流水线。

| 技能 | 作用 |
|---|---|
| `0-init-project-context` | 初始化/刷新 `.context/`，提炼项目背景、架构与规则基线 |
| `1-requirements-analysis` | 逐步澄清需求，产出可落地的需求分析，经独立评审把关 |
| `2-technical-design` | 把需求转成技术设计（方案/分层/数据库/接口），经独立评审 |
| `3-writing-plan` | 把设计拆成有序、可验证的实现步骤，内置分层 TDD 策略 |
| `4-executing-plan` | 按计划落地编码，验证通过后回写进度，汇总执行报告 |
| `5-committing-changes` | 质量清单检查、过程文档归档提炼、按项目惯例提交 |
| `6-retrospective` | 把坑与约定提炼回规则库/经验库，让项目越用越聪明 |
| `dev` | 功能开发统一入口，按任务规模路由到极简/快速/标准/完整 |
| `bugfix` | 轻量修复线：复现 → 根因 → 分流 → 修复 → 验证 → 沉淀 |

技能会被 Claude 根据上下文**自动触发**；也可显式调用，如 `/ctxloom:dev`、`/ctxloom:bugfix`、`/ctxloom:2-technical-design`。

## 安装

```bash
# 1. 添加本市场（GitHub）
/plugin marketplace add lihu-001/ctxloom

# 也支持完整 git 地址：
# /plugin marketplace add https://github.com/lihu-001/ctxloom.git

# 2. 安装插件
/plugin install ctxloom@ctxloom
```

安装时可选作用域：`User`（个人全局，跨所有项目）、`Project`（写入 `.claude/settings.json`，随仓库共享给协作者）、`Local`（仅当前项目、仅自己）。

## 使用案例

技能由 Claude 按你的话自动触发，下面的对话直接用日常说法即可，无需记命令。

### 案例 1：第一次接手一个项目

```
你：帮我熟悉一下这个项目并建立上下文
```

`0-init-project-context` 启动：检索代码（项目配了 CodeGraph 就优先用它），把技术栈、分层、模块协作写进 `project.md`，从现有代码归纳出通用规则填进 `rules/`，并把构建/测试命令、安全高危点固化下来。跑完一次，后续每个技能都有统一的项目背景可读。

```
.context/ 已建立：
  project.md      — Spring Boot + MyBatis，3 个模块，分层与数据流已提炼
  rules/build.md  — mvn -pl order compile
  rules/testing.md— JUnit5，mvn -Dtest=XxxTest test
  rules/security.md — 已标注 SQL 拼接、${} 直插等高危点
  其余规则文件、活文档目录、README 均已就位
```

### 案例 2：开发一个新功能（dev 自动选路）

```
你：给订单加个导出 Excel 的接口
```

`dev` 启动，先做规模评估，告诉你判断依据和选哪条路：

```
判断：方案基本唯一、改动集中在 2-3 个文件、需求清楚、风险低、无架构影响
→ 走「快速路径」

实现方案：在 OrderController 加 GET /orders/export，
复用 OrderService.query 取数，用 EasyExcel 写流式响应。
确认就开始改。
```

你点头后它才动代码，改完就地验证并提示用 `5-committing-changes` 收尾。如果这是个跨多模块、要权衡方案的大功能，dev 会自动升级到标准或完整路径，依次调用需求分析、技术设计、计划、执行。

只抛一个名词、意图不明时，dev 不会默认开干，会先一句话确认：

```
你：用户积分
dev：你是想直接开发「用户积分」功能，还是先理清需求（暂不写代码）？
```

### 案例 3：修一个线上 bug

```
你：线上提交订单后金额算成 0 了，帮我查一下
```

`bugfix` 启动，先坐实复现、用证据定位根因，再按复杂度分流：

```
复现：选中带优惠券的订单 → 提交 → 金额为 0（必现）
根因（有证据）：couponBranch 分支下 rate 字段未赋值，
  导致 amount * rate = 0。日志与最小复现均指向此处。
分流：改动局部、根因明确 → 简单修复，最小改动赋默认 rate=1。
验证：重跑复现，金额正确；补了一个覆盖该分支的单测。
```

复杂 bug 它会起独立智能体复核根因，并升级复用 `2`/`3`/`4`。

### 案例 4：完整流程跑一个复杂功能

需求模糊、跨多模块的大功能，dev 会选完整路径，依次串起 0–6：

```
你：做一套优惠券发放与核销系统
dev：判断为完整路径（需求待澄清、跨多模块、有架构影响）

1-requirements → 澄清边界，产出 requirements/coupon.md，独立评审把关
2-design       → 方案选型/分层/库表/接口，designs/coupon.md，独立评审
3-plan         → 拆成有序可验证步骤，tasks/coupon.md
4-execute      → 子智能体按步落地，按测试策略验证，回写进度
5-commit       → 质量清单 + 把活文档提炼归档到 archive/coupon.md + 提交
6-retro        → 把踩的坑和定的约定回写 rules/ 与 learnings.md
```

同一个功能从 `requirements/coupon.md` → `designs/coupon.md` → `tasks/coupon.md` → `archive/coupon.md` 用同名文件贯穿，顺着就能看清全貌。
