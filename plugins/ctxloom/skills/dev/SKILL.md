# 开发任务入口

本技能是所有功能开发的入口。它的使命：**让任务默认走完整流程，再按复杂度调节每一步的深浅，而不是靠跳过步骤来省事**。

设计取舍：这套技能的核心价值，就在于"需求→设计→计划→执行→提交→复盘"这条完整流程能把功能想清楚、做扎实。所以默认就走完整流程——简单任务也走，只是每步轻量带过；复杂任务也走，每步充分展开。**减负靠的是"每步缩放"，不是"整段跳过"**。唯一的例外是极琐碎的改动（改错别字、调一个常量），走逃生口直接做。

## 整体流程

完整流程图见原版 SKILL.md 或 `specs/dev-workflow.md`。核心步骤：

1. **检查 `.ctxloom/`**：不存在时询问是否初始化
2. **意图确认**：用户输入模糊时确认是"开发"还是"理清需求"
3. **极简逃生口粗筛**：四条全中才走极简档（方案唯一、≤10行、单文件、低风险）
4. **定档**：起独立子智能体定基线档（standard/deep）+ 给依据 + 反向证据
5. **走完整流程**：1→2→3→4→5→6（每步按档位缩放）

## 定档原则

通过独立子智能体评估任务复杂度，定为 **standard** 或 **deep** 两档：

**标准档（standard）**：
- 需求明确、边界清晰
- 方案直接、无需多方案权衡
- 改动面窄（单模块、少量文件）
- 低风险（不涉及核心流程、无数据迁移）

**深度档（deep）**：
- 需求模糊、边界不清或有争议点
- 方案有多种选择、需权衡
- 改动面广（跨模块、多文件）
- 高风险（涉及核心流程、数据迁移、安全敏感）

**极简档（极简逃生口）**：
- 只有四条**全部满足**才走极简：
  1. 方案唯一——只有一种显然做法
  2. 代码量极少——预计改动 ≤10 行
  3. 单文件改动——不跨文件、不牵连其他模块
  4. 低风险——不涉及核心流程、无数据变更、无安全风险

## 档位写入与传递

定档后，将档位写入任务活文档 `.ctxloom/tasks/<功能>.md` 的 frontmatter：

```yaml
---
type: Task
feature: <功能>
grade: standard  # 或 deep
status: in-progress
...
---
```

下游技能（1-6）在执行时读取这个 `grade` 字段，按档位缩放自己的执行深度。

## 极简逃生口流程

四条全中时：
1. 一句话说要改什么
2. 改代码（直接用 Edit/Write）
3. 告知改动与验证建议（怎么测、注意什么）
4. 进入 6-retrospective 复盘

## 标准/深度档流程

调用下游技能（用 Skill 工具，带完整限定名 `ctxloom:<技能名>`）：

```
1. ctxloom:1-requirements-analysis
2. ctxloom:2-technical-design  
3. ctxloom:3-writing-plan
4. ctxloom:4-executing-plan
5. ctxloom:5-committing-changes
6. ctxloom:6-retrospective
```

每个技能会自己读取任务活文档的 `grade` 字段并按档位缩放。

## 前置检查

### 第 0 步：`.ctxloom/` 检查

**立即检查 `.ctxloom/` 目录是否存在**：

- **存在** → 继续
- **不存在** → 问用户："项目还没初始化上下文目录（`.ctxloom/`），要先初始化吗？"
  - 同意 → 先说明"这是与你这个需求无关的整库初始化"，然后用 Skill 工具调用 `ctxloom:init`
  - 拒绝、坚持直接开发 → 可继续，但提醒建议仅走极简档

### 第 1 步：意图确认

如果用户只抛出业务名词、没动词（如"订单导出""用户积分"），先确认是"直接开发"还是"先理清需求"：

- 直接开发 → 继续极简粗筛
- 先理清需求 → 转交 `ctxloom:1-requirements-analysis`，本技能退出

## 代码探索原则

需要检索代码时：

- **CodeGraph 可用时结构性查询必须用它**：`codegraph_context`/`codegraph_search`/`codegraph_node`/`codegraph_callers`
- **grep/Read 只用于查字面文本或已定位文件后的细看**
- **只有 CodeGraph 确实未启用时才退回 grep/读文件兜底**
