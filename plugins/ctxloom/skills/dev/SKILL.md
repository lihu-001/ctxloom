# 开发任务路由

本技能是所有功能开发的入口。它的唯一使命：**在开始写代码之前，先判断这个任务适合哪条路，然后按那条路走，不多也不少**。

简单任务套完整流程会让人烦；复杂任务跳过分析直接写会让人踩坑。本技能通过一次快速评估，把任务导入合适的轨道。

## 路径评估

快速在脑子里过这 5 个问题：

| 问题 | 倾向简单 | 倾向复杂 |
|---|---|---|
| 方案一目了然？ | 是，只有一种显然做法 | 需要调研或权衡多方案 |
| 改动有多大？ | ≤10 行，单文件或单模块 | 多文件/跨模块 |
| 需求清楚吗？ | 用户说清楚了 | 有歧义，需要澄清 |
| 风险高吗？ | 低，容易回退 | 影响核心逻辑/向后兼容 |
| 有架构影响吗？ | 无，局部改动 | 涉及接口设计/数据模型/跨系统 |

根据倾向简单的问题数量选路径：

- **5 个** → 极简路径
- **3-4 个** → 快速路径
- **2-3 个** → 标准路径
- **0-1 个** → 完整路径

有疑问时宁可往上走一档。选定后，用一句话告知用户走哪条路，然后立刻执行。

---

## 极简路径

**适用**：方案唯一、改动 ≤10 行、单模块、低风险

直接做三件事：

1. 一句话说明要改什么（"我看到是 X，改 Y"）
2. 改代码
3. 告知改了什么，以及建议怎么验证

没有文档、没有计划、没有流程——改完就交付。改着改着发现比预期复杂，随时升级到快速路径。

---

## 快速路径

**适用**：需求清楚、2-3 个文件、有少量设计决策、低中风险

### 1. 确认要点（对话内完成）

用 2-3 句话复述对需求的理解，明确边界和关键决策点。有歧义就问，但只问影响实现的核心问题，边角细节先用合理默认推进。用户确认后立即执行。

2 轮内对齐不了核心要点，主动升级到标准路径，不硬撑往下走。

### 2. 执行

直接在当前工作区改代码，不建计划文档。执行规范：

- **只改与需求直接相关的代码**，不顺手重构、不夹带
- **改完就地验证**：能跑的跑一下，不能跑的说清楚怎么验
- **有测试框架的项目**，视情况补 1-2 个关键用例；没有则跳过
- **遇到意外**（改一处又发现别处问题、改动超出预期）：停下来同步用户，确认是否升级路径

### 3. 收尾

告知改了哪些文件、做了什么、建议怎么验证。提示用户可用 `5-committing-changes` 做提交归档。

## 标准路径

**适用**：中等复杂度、跨 2-3 个模块、需要设计决策、有一定风险

**前置检查**：先看 `.context/` 是否存在。
- **存在** → 直接进入下面的流程
- **不存在** → 提示用户先用 `0-init-project-context` 初始化项目上下文，再回来继续；用户若明确说不需要，可降级到快速路径执行

流程：

1. 用 Skill 工具调用 `1-requirements-analysis`（告知用户做简化版，重点澄清核心边界，不做完整的六步评审）
2. 根据改动面决定是否需要 `2-technical-design`：单一明确方案可跳过，有多个方案需权衡时调用
3. 用 `4-executing-plan` 执行（无正式计划文档时，在任务活文档里列简要步骤作为执行依据）
4. 提示用户用 `5-committing-changes` 提交归档

跳过：`3-writing-plan`（改动面不大，步骤心里有数，不需要展开成完整计划）

---

## 完整路径

**适用**：需求模糊、跨多模块、架构影响、高风险、方案权衡复杂

依次调用：`1-requirements-analysis` → `2-technical-design` → `3-writing-plan` → `4-executing-plan` → `5-committing-changes` → `6-retrospective`

各阶段按各自技能的规范执行，本技能不重复定义。

---

## 流程回溯规则（所有路径通用）

在任意节点发现问题，若该问题涉及上游节点的产物（需求文档、设计文档、实现计划），必须回到对应的上游节点调整，不能绕过或就地打补丁——打补丁会让上下游文档逐渐失真，变成没人信的死文档。

**回溯矩阵**：

| 问题性质 | 回到哪里 |
|---|---|
| 需求不合理 / 遗漏场景 / 需求矛盾 | `1-requirements-analysis`（调整需求文档） |
| 技术方案不可行 / 有更优方案 / 依赖关系变化 | `2-technical-design`（调整设计文档） |
| 实现步骤需要拆分/合并/重排 | `3-writing-plan`（调整计划文档） |
| 已写代码需要修改 | `4-executing-plan`（重新执行受影响步骤） |

**发现问题时，先起子智能体裁定，再告知用户**：

主智能体在执行上下文里容易偏向"过度回溯"或"忽略问题"。发现疑似需要回溯的问题时，用 Agent 工具起一个独立子智能体做裁定：

给子智能体的输入：
- 发现的问题描述（现象是什么、为什么觉得有问题）
- 当前处于哪个节点
- 相关的需求/设计/计划文档片段（如有）

要求子智能体独立判断并返回：
1. **是否需要回溯**：真的需要回到上游调整，还是可以在当前节点就地处理？
2. **回溯到哪个节点**（需要回溯时）：1 / 2 / 3 / 4，以及理由
3. **建议的调整范围**：哪些部分需要改，哪些不用动

拿到裁定结果后，告知用户："发现了 X 问题，独立判断建议回溯到节点 Y，调整范围是 Z，是否确认？"用户确认后再动。

**回溯时的原则**：

- **调整而非重建**：以已有文档为基础，只修改受影响的部分，不清空重来
- **级联更新**：从回溯点往下，所有受影响的下游文档同步更新（需求改了→设计跟着调→计划跟着调）
- **注明原因**：修改文档时注明为什么改，避免将来看不懂这里为什么有这个调整

## 所有路径通用：经验沉淀

**任意路径完成后，必须调用 `6-retrospective`。** 由它判断本次有无值得沉淀的经验——有就记，没有就直接结束，不强凑。完整路径已将其纳入流程；极简/快速/标准路径在交付后立即调用。
