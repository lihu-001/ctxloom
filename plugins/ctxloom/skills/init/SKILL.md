## 概述

本技能为项目建立一套标准化的「项目上下文」目录，让后续的 AI 协作（需求分析、设计、编码、评审）有统一的背景知识和规则可依据。

核心产物是项目根目录下的 `.ctxloom/` 目录，它把几类信息固化下来：

- **项目背景与架构** —— 这个项目是做什么的、怎么分层、关键模块如何协作（落点 `project.md`）
- **项目规则** —— 代码风格、编码约定等团队规则，必须是**通用规则**而非针对某个文件的描述；其中**构建命令**、**测试方式**、**安全高危点**三类信息是下游技能(计划/执行/提交)落地验证与安全检查的强依赖，本技能要专门固化下来（分别落 `rules/build.md`、`rules/testing.md`、`rules/security.md`）
- **长期记忆** —— 踩坑经验直接沉淀进 `rules/`（带背景注），不再单设 `learnings.md`。本项目历史需求通过 `requirements/` 下的文档与索引自然承载，不再单设台账
- **过程文档（活文档）** —— 每个功能一份贯穿需求/设计/执行的活文档（`requirements/`、`designs/`、`tasks/`），同名文件贯穿全流程

> **语言无关原则。** 这套 0–6 技能要跨 Java、Vue 等多种项目复用，因此技能定义本身只写语言无关的方法论；一切与语言/项目相关的具体内容(构建命令、测试框架与命令、安全高危模式、提交规范)都固化在本项目的 `.ctxloom/rules/` 里，由下游技能按需读取。这就是为什么本技能必须把「构建命令」「测试方式」「安全高危点」固化清楚——下游技能不写死 `mvn`/`npm`，而是来这里读。

## 硬性规则

- 规则文件里写的规则**必须通用**。规则描述的是"在这个项目里应当怎么做"，适用于未来任意新代码，而不是"某个类现在是怎么写的"。这是本技能最重要的约束——一条只对单个文件成立的描述不是规则。
- 不臆造项目背景与架构。只写从代码、配置、文档中**实际检索到**的信息；无法确认的部分明确标注「待补充」，不要编造。
- 已存在的内容默认保留并补全/刷新，不要无差别覆盖用户已经手写的内容（详见「老项目：补全与刷新」）。
- 项目背景/架构以 `.ctxloom/project.md` 为**权威来源**。项目若已有 `CLAUDE.md`，把它当作检索信号源即可，不要把长篇背景/架构同时塞进 `CLAUDE.md` 与本目录而形成两套并存、各自漂移的描述（详见「与 CLAUDE.md 的关系」）。
- **每份文档自描述身份、消费时宽容。** `.ctxloom/` 是一个自描述的知识目录：每个 `.md`（`README.md` 除外）头部带一段 YAML frontmatter 标明它的 `type` 等元信息（详见 `specs/frontmatter.md`）。但**读取**这些文档时必须**宽容**——容忍缺 frontmatter、缺字段、未知 `type`、坏的交叉链接，绝不因格式不全而报错或拒绝处理。这条对本技能与下游 1–6 同样适用：frontmatter 是叠加增强，缺了也要能照常按路径读正文。

## 上下文管理原则

第 2、3 步的项目检索是**全项目多维度普查**，会积累大量上下文。采用两个杠杆防范风险：

- **探索委派**（防膨胀）：把摸代码搬到子智能体的独立上下文里，主上下文只收提炼结论
- **阶段性落盘**（防丢失）：每完成一阶段立即写盘，压缩后可从磁盘读回

### 探索委派模型

主智能体在第 2 步**只当「指挥 + 落盘」**，派发 4 个探索子智能体（协议见 `agents/` 下各 `AGENT.md`），子智能体只回传提炼结论，原始源码不进主上下文。

四个探索子智能体（**彼此无依赖，一次并行全发**）：

| Agent | 维度 | 落点 |
|---|---|---|
| `stack-explorer` | 技术栈 + 模块结构 + 整体架构 + 外部依赖 | `project.md` |
| `convention-explorer` | code-style + coding + other 通用规则 | `rules/code-style.md`、`coding.md`、`other.md` |
| `buildtest-explorer` | 构建打包命令 + 测试框架与命令 | `rules/build.md`、`testing.md` |
| `security-explorer` | 本技术栈真实高危模式 | `rules/security.md` |

**派发要点：**

- 用 Agent 工具起 `general-purpose` 子智能体，prompt 里贴上对应 `AGENT.md` 的协议
- 提供：项目根路径、**CodeGraph 是否可用（主智能体在第 2 步开头探测一次，传给所有子智能体）**
- 四个 explorer 在同一条消息里一次并行发出（多个 Agent 调用）
- 子智能体回传的是不带 frontmatter 的正文；主智能体落盘时按规范补头部

> **与全局「不要委派探索」口径的关系。** 单个聚焦问题（how does X work）在主上下文里用 codegraph 直接答最省；但 init 是**整库普查**，聚合量超预算，委派才对。

### 落盘检查点

**强制落盘节点**（每完成一个阶段立即落盘）：

1. ✅ **完成第 1 步后**：目录结构与空文件创建完毕，立即落盘
2. ✅ **完成第 2 步后**：四个 explorer 回传齐了，`project.md` 与所有 `rules/*.md` 提炼完毕，立即全部写入磁盘
3. ✅ **完成第 3 步后**：`current.md`、各 `README.md` 全部写入磁盘

**上下文压力信号**（满足任一条件立即落盘）：
- 已调用 codegraph/Read 工具超过 30 次
- 已读取的文件内容明显很长（单个文件 > 500 行，或累计读取 > 3000 行）
- 感受到对话历史已经很长

## 目录结构

`.ctxloom/` 的标准结构详见 `specs/directory.md`。核心要点：

- 带 `.md` 后缀的是文件，其余都是文件夹
- 除 `README.md` 外，每个 `.md` 文件头部都带 frontmatter（规范见 `specs/frontmatter.md`）
- `requirements/`、`designs/`、`tasks/` 下的 `<功能>.md` 是**活文档**，同名文件贯穿全流程
- 活文档之间用交叉链接互联（规范见 `specs/linking.md`）

> **目录契约的单一来源。** `specs/directory.md` 是整套 0–6 技能流程的**权威定义**。后续技能会引用其中的具体路径。若要调整结构，**以那里为准**，并同步更新引用这些路径的技能。

## Frontmatter 与交叉链接

- **Frontmatter 规范**：详见 `specs/frontmatter.md`，写文档时按规范填，读文档时宽容处理
- **交叉链接约定**：详见 `specs/linking.md`，活文档开头放关联指引行，消费方容忍坏链

### 第 1 步：检测上下文目录是否存在

检查项目根目录下是否已有 `.ctxloom/`。

- **不存在** → 走「新建」分支：按 `specs/directory.md` 的结构创建全部目录与文件（每个 `.md` 按 `specs/frontmatter.md` 写好头部），然后把目录与文件清单写入 `README.md`（见「README.md 的内容」）。
- **已存在** → 走「补全与刷新」分支（见「老项目：补全与刷新」）：逐项比对结构，补建缺失的目录/文件，并刷新 `project.md` 与规则文件的内容。

无论哪个分支，最终都要保证目录结构完整、`README.md`/`project.md`/规则文件都有内容。

### 第 2 步：探测 CodeGraph，一次并行派 4 个探索子智能体，落盘 project.md 与规则文件

#### 2.0 空仓短路（全新项目先判一道）

派 explorer 做重普查之前，先判断项目是不是**近乎空仓**——还没什么源码可查（典型是刚 `git init`、只有 README/配置脚手架、源码文件寥寥无几）。判断信号：源码文件极少、没有成形的分层结构、构建配置也只是模板默认值。

- **近乎空仓** → **不必并行起 4 个 explorer 做重普查**。直接：
  - 第 1 步已建好的目录骨架保留
  - `project.md`、`rules/*.md` 全部以「待补充」+「基于通用最佳实践的占位」落盘（仍按 `specs/frontmatter.md` 补头部），不强行编造项目特有信息
  - 第 4 步告知用户：「项目尚无足够代码，已建好空白上下文基线（规则暂为通用占位），随开发推进会在后续需求路过相关模块时自动补全。」
  - **跳过 2.1 / 2.2**，直接进第 3 步
- **已有成形代码** → 正常往下走 2.1 / 2.2 的整库普查

> **边界守住：这是「空仓豁免重普查」，不是「按需求聚焦初始化」。** 空仓短路只在确实没代码可查时省掉 explorer，省的是「查不出东西的徒劳普查」；一旦项目有成形代码，仍是与当前需求无关的**整库**普查，绝不收窄到当前需求相关的模块。

#### 2.1 探测 CodeGraph 可用性（一次，供所有 explorer 共用）

尝试调用一次 `codegraph_status`：

- **能正常返回** → CodeGraph 可用。记录这一结论，所有探索子智能体都会收到「CodeGraph 可用」信号。
- **报「未初始化 / 工具不可用」** → CodeGraph 缺失，按缺哪层处理（**必须征询用户、绝不静默安装**）：
  - **项目无 `.codegraph/` 索引**（最常见）→ 提示用户：「这个项目还没建 CodeGraph 索引，建好后本次以及后续需求/设计/计划阶段的代码检索都会更快更准、更省上下文。要我建吗？（你也可自行执行 `codegraph init -i`）」。**得到用户同意再建**；不同意 → 记录「CodeGraph 不可用」。
  - **连 `codegraph` 命令都没有** → 只提示一句安装方式（全局安装 codegraph CLI），由用户自行决定，**不要静默 `npm install -g`**。记录「CodeGraph 不可用」。
- **兜底不阻塞**：CodeGraph 不可用时，所有探索子智能体会按其协议退回 grep/逐文件读兜底，照常完成本技能全部产出；并在第 4 步汇报里注明「本次未启用 CodeGraph，建议后续初始化以提升各阶段检索效率」。

#### 2.2 一次并行派发 4 个探索子智能体

技术栈/架构、编码约定、构建测试、安全高危四个维度彼此无依赖，**在同一条消息里一次并行发出 4 个 `general-purpose` 子智能体**（多个 Agent 调用）。

| Agent | 协议路径（贴进 prompt） | 回传 → 落盘 |
|---|---|---|
| `stack-explorer` | `agents/stack-explorer/AGENT.md` | 技术栈/架构 → `project.md` |
| `convention-explorer` | `agents/convention-explorer/AGENT.md` | code-style / coding / other 三块 → `rules/code-style.md`、`coding.md`、`other.md` |
| `buildtest-explorer` | `agents/buildtest-explorer/AGENT.md` | build / testing 两块 → `rules/build.md`、`testing.md` |
| `security-explorer` | `agents/security-explorer/AGENT.md` | 安全清单 → `rules/security.md` |

每个子智能体的 prompt 统一格式（贴协议 + 项目根路径 + CodeGraph 可用性）：

```
你现在作为<某维度>探索子智能体。协议如下：

<贴上对应 AGENT.md 的全部内容>

---
任务输入：
- 项目根路径：<绝对路径>
- CodeGraph 可用性：<可用 / 不可用>

请按协议探索并回传<对应文件>正文（不带 frontmatter）。
```

四个子智能体回传各自的提炼结论（均为不带 frontmatter 的正文）后，主智能体统一落盘：

1. **`project.md`**：按 `specs/frontmatter.md` 补 `type: Project` 等头部，Write。
2. **`rules/*.md`**：按 `specs/frontmatter.md` 给每个规则文件补头部（`type: Rule`、`build`/`testing` → `downstream: [3, 4, 5]`，`security` → `downstream: [2, 5]`），立即 Write 全部。

**为什么规则必须通用**（落盘前主智能体对子智能体回传的规则做一次把关）：每条规则都应能直接套用到一段还没写的新代码上。自检：「明天新增一个类，这条规则能指导它怎么写吗？」能，才是规则。

```markdown
# 反例（不要这样写——这是对现状的描述，不是规则）
- UserServiceImpl 里用了 @Transactional

# 正例（通用规则）
- Service 实现类中涉及多次写库的方法统一加 @Transactional，保证原子性
```

### 第 3 步：建立过程文档索引

> **历史需求索引不再单设台账。** 「本项目历来做过哪些功能」由 `requirements/README.md` 功能索引自然承载，通过文档的 `status` frontmatter 字段区分在做/已交付。本技能新建时把 `README.md` 建好（从 `specs/templates/requirements-readme.md` 读取模板）；老项目刷新时，尽量从已有过程文档回填索引行。

### 第 4 步：完成后告知用户

简要汇报：建立/补全了哪些目录与文件、背景架构提炼了哪些要点、规则文件各写入了几条规则、各文档是否已按规范带上 frontmatter、哪些地方标了「待补充」或基于通用最佳实践需要团队校准。

## 老项目：补全与刷新

当 `.ctxloom/` 已存在时（典型是老项目第二次运行本技能）：

1. **结构补全**：逐项对照 `specs/directory.md`，缺失的目录和文件补建出来；已存在的目录/文件保留。
2. **内容刷新**：重新检索项目，刷新 `project.md` 与规则文件——**刷新同样走「探索委派模型」**，但**合并由主智能体做、不要无差别覆盖用户手写的内容**。做法是：
   - 子智能体回传「从当前代码探到的最新事实」后，主智能体把它与磁盘上的现有文件**对比合并**
   - 若文件内容明显是本技能上次生成的，可整体重写为最新版本
   - 若文件中包含用户后来补充/修改的内容，保留这些内容，只在其基础上补充新发现的信息
3. `README.md` 中的清单同步更新为当前实际结构。
4. **回填历史索引**：扫描 `requirements/`、`designs/`、`tasks/` 下已有的活文档，把 `requirements/README.md` 的功能索引补齐。
5. **补建规则强依赖文件**：老项目若缺 `rules/build.md`、`rules/testing.md`、`rules/security.md`，按第 2 步的要求新建——这是下游技能跨语言运行的强依赖。
7. **从旧结构迁移（如适用）**：若项目残留早期版本的 `.项目上下文/` 目录，提示用户这是旧结构，建议迁移到新结构，**得到用户同意再做**。
8. **升级 frontmatter 与交叉链接**：老项目的既有 `.md` 若缺 frontmatter，按 `specs/frontmatter.md` **补写**；活文档若缺开头的关联指引行，按 `specs/linking.md` 补一行。

## 与 CLAUDE.md 的关系

项目背景、架构、技术栈这些信息，在 `CLAUDE.md` 与 `.ctxloom/project.md` 里容易各写一份、日久漂移。两者分工应当明确：

- **`.ctxloom/project.md` 是权威、详尽的来源。** 它按需读取，适合放完整的背景与架构。
- **`CLAUDE.md` 保持精简、只留指针。** 它会被每个会话自动载入，长篇背景放这里既费 token 又与本目录重复。

据此，本技能的处理：

1. **检索阶段**：把已有的 `CLAUDE.md` 当作信号源读取，但项目背景/架构的最终落点是 `project.md`。
2. **补指针（征得同意）**：若项目已有 `CLAUDE.md` 却没有指向 `.ctxloom/` 的指引，建议补一行短指针——**先征得用户同意再加**。项目若没有 `CLAUDE.md`，本技能不替它新建。
3. **刷新阶段对账**：老项目刷新时，若发现 `CLAUDE.md` 里的背景/架构与 `project.md` 不一致，不要默默以某一方覆盖另一方，把差异点摆给用户。

## README.md 的内容

`.ctxloom/README.md` 是整个上下文目录的导览，也是这套 frontmatter / 链接规范在仓库里的**内置副本**。它需包含：

1. **目录与文件清单** —— 从 `specs/directory.md` 读取完整目录树
2. **每个文件/目录的用途说明** —— 逐项说明它是干什么用的
3. **Frontmatter 规范** —— 从 `specs/frontmatter.md` 读取**完整规范**写入 README
4. **交叉链接约定** —— 从 `specs/linking.md` 读取写入 README
5. **使用约定 / 工作流说明** —— 说明这套目录如何配合开发流程使用

`.ctxloom/README.md` 是本目录里**唯一可带 frontmatter 的 `README.md`**，且仅允许 `okf_version: "0.1"` 一项。

写 `README.md` 时用平实的语言，让一个第一次看到这个目录的同事能据此正确使用它。
