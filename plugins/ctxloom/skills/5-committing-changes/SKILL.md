# 提交改动与归档文档

本技能承接 4-executing-plan 的收尾——4-executing-plan 明确"不替用户碰 git"，把代码的最终把关权和提交权留给用户。本技能就是用户决定提交时接过这一棒：把一次已经验证通过的开发成果，连同它的过程文档一起干净地落库。

它的价值在于**让收尾这件事标准、可靠、不出错**：代码质量在落库前先过一道清单门禁、提交信息符合项目惯例、过程文档该归档的归档、敏感文件不被误提交、不污染主干。这些事零散去做容易漏，固化成一条流程就稳了。

> **语言无关原则。** 本技能跨 Java、Vue 等多种项目复用，因此**不写死任何语言专属内容**。代码质量检查的具体高危模式(如何种 SQL 注入写法、何种 XSS、何种敏感配置)一律读本项目 `.ctxloom/rules/security.md` 里固化的「安全高危点」与 `.ctxloom/rules/coding.md`；构建/测试验证命令读 `.ctxloom/rules/build.md` 与 `.ctxloom/rules/testing.md`。技能本身只描述语言无关的检查维度与流程。

## 硬性规则

这几条是底线，任何时候都不能违背：

- **提交前必须让用户过目确认。** 提交是把改动正式落库的动作，提交了什么、提交信息怎么写，要先摆给用户看、得到确认再执行 `git commit`。不要摸清改动后直接闷头提交。这道确认是用户行使把关权的地方。
- **提交前必须做代码质量清单检查，门禁未过不擅自提交。** 提交前按步骤 2 的规则对本次**源代码改动**逐条检查，列出问题清单并按 **阻断 / 重要 / 建议** 三级标注，把质量报告交给用户。**无阻断、无重要问题** 才可顺势进入提交确认；**存在任何阻断或重要问题**，一律**停下来让用户选择**——是先修问题再重查，还是知悉风险后仍要照常提交——得到用户明确选择再继续，不替用户决定、不带着未决的阻断/重要问题闷头提交。
- **不直接提交到主干。** master/main 是受保护的主干。除非用户明确要求提交到当前主干分支，否则提交应落在功能分支上（通常 4-executing-plan 已经切到了形如 `feat/xxx` 的分支，顺着用就行）。若发现当前正处于 master/main 且用户没明说要提到主干，停下来与用户确认分支。
- **敏感文件不入库，发现就提示。** 提交前扫一眼待提交清单，如出现 `.env`、密钥文件、凭证、`*.pem`、本地配置等可能含密的文件，先提示用户、确认是否真的要提交，而不是默默 `git add` 进去。(项目规则里若写明了敏感配置的存放约定，以其为准。)
- **push 是对外动作，必须先问。** push 会把改动发到远端、被他人看到，不可轻易撤回。本技能提交后**只询问**用户是否推送，得到明确同意才执行 `git push`；用户没点头就停在本地提交这一步。
- **归档是提炼不是复制。** 功能交付时，把该功能的需求/设计/执行活文档**提炼归纳总结**成一份精炼的 `archive/<功能>.md` 成品——它是"这个功能最终做成了什么"的浓缩结论（最终方案、关键决策、与初始设想的偏差、遗留项），不是把活文档原样搬过去。活文档本身留在 `requirements/`、`designs/`、`tasks/` 原处作详细过程记录，不删除、不移动，仅把它们 frontmatter 的 `status` 翻成 `delivered` 以示封存。

## 工作流程

> **开始前——关于项目上下文地基。** 本技能会把本次功能的活文档**提炼归档**成 `.ctxloom/archive/<功能>.md`。若项目根本没有 `.ctxloom/`（例如纯代码改动、未走过 0-6 流程），归档无从谈起，跳过归档步骤、只做提交即可。若有 `.ctxloom/` 但缺 `archive/` 目录，按步骤 4 在写入前先创建。
>
> 本技能引用的 `.ctxloom/` 路径以 `0-init-project-context` 的《目录结构》为权威定义，结构调整以其为准。

### 步骤 1：摸清本次要提交什么

先把当前仓库状态和本次改动看清楚，这是后面写提交信息、判断归档范围的基础：

- 跑 `git status` 和 `git diff --stat`（必要时 `git diff` 看具体改动），弄清楚改了哪些文件、属于哪个模块、大致是什么性质的改动（新功能 / 修复 / 重构 / 文档）。
- 看一眼当前在哪个分支（`git branch --show-current`）。若在 master/main 上，按硬性规则先与用户确认分支问题。
- 识别本次需求是什么。**先读 `.ctxloom/tasks/current.md`**：其中记录的当前在途功能（阶段已到「执行已完成、待提交归档」）通常就是本次要收尾的需求。`current.md` 缺失或与实际对不上时，再从改动涉及的模块、以及 `.ctxloom/tasks/` 下进行中的活文档名推断（如改动集中在某模块、恰有一份对应该模块的活文档，那本次需求多半就是它）。

这一步你自己做功课，把"改了什么、在哪个分支、对应哪个需求"心里有数。

### 步骤 2：代码质量清单检查

在写提交信息、正式提交之前，对本次**源代码改动**做一次质量把关——此时 diff 刚在手里，是天然的检查点。把好关再落库，比提交后回头返工省事得多。**本步用清单式检查 + 三级问题门禁，不打数字分**（数字分容易凭印象给、也容易为了"够线"而注水；逐条列出真实问题、定级处理，更可靠也更可复核）。

**检查范围：只看源代码改动**，即本次改动中承载业务逻辑的源码文件（按项目技术栈而定：`.java`、`.js`、`.ts`、`.vue`、`.py`、`.go` 等）。文档（`.md`）、归档移动的文件、构建产物、`.claude/`、`.omc/` 等状态/配置文件**不纳入检查**。若本次改动里没有任何源代码（例如纯文档提交），跳过本步骤、在报告里说明"本次无源代码改动，跳过质量检查"，直接进入后续流程。

**先读项目规则再检查。** 检查的依据是本项目 `.ctxloom/rules/` 里固化的约定——尤其 `security.md` 的「安全高危点」和 `coding.md`。语言相关的高危模式(哪种 SQL 拼接、哪种 XSS、哪种敏感配置)以那里写的为准，本技能不写死。

**逐项过 diff 检查。** 把上一步看到的源码改动逐个文件、逐段过一遍，按下面六个语言无关的维度审视，发现问题就记下来（`文件:行` + 问题 + 怎么改）：

| 维度 | 主要考察点 |
|---|---|
| 功能正确性与逻辑 | 是否实现预期意图；逻辑/取数错误（如查询的过滤、分组、聚合、排序条件）；边界与空值（空引用、空集合、越界）；并发安全；off-by-one 等 |
| 安全性 | 注入类风险（拼接式查询、模板直插）；硬编码密钥/密码/地址；缺输入校验；越权与敏感信息泄露到日志/前端——**具体高危模式按项目规则的「安全高危点」核对** |
| 健壮性与错误处理 | 异常是否被吞掉/裸捕获不处理；资源是否释放；失败是否有回退或明确报错；关键路径有无必要日志 |
| 代码规范与可读性 | 命名清晰；与项目既有风格(`.ctxloom/rules/`)一致；注释恰当不冗余；无注释掉的死代码、无调试输出残留 |
| 设计与可维护性 | 有无可消除的重复代码；职责是否单一；抽象是否合理；有无与本次需求无关的过度设计 |
| 验证与测试 | 是否能按 `.ctxloom/rules/build.md`/`testing.md` 的命令编译/构建通过；关键路径是否已验证；新功能/修复有无相应测试，test-first 步骤是否留有先红后绿的痕迹（项目测试薄弱时，至少说明验证方式） |

**给每个问题定级**（这是门禁的依据）：

| 级别 | 含义 | 处理 |
|---|---|---|
| 阻断 | 功能错误 / 安全漏洞 / 编译构建不过 / 会导致线上故障 | 必须修，不修不提交 |
| 重要 | 不影响能跑，但有明确隐患或明显坏味道、规则偏离 | 停下让用户选：修 or 知风险照提 |
| 建议 | 可优化项、风格不一致、表述不清 | 可顺手改，也可记录后照提 |

**门禁判定**（对应硬性规则）：

- **无阻断、无重要**：质量过关，输出报告后顺势进入步骤 3。
- **有阻断或重要**：输出报告、列清问题，**停下来让用户选**——先修再重查，还是知悉风险后照常提交。阻断问题要明确表达"强烈建议先修"的倾向。

**质量检查报告模板**——按这个结构输出给用户：

```
## 代码质量检查报告

**门禁结果：<通过 ✅ ｜ 存在阻断/重要，待用户决定 ⚠️>**

### 问题清单
| 级别 | 位置 | 问题 | 建议 |
|---|---|---|---|
| 阻断 | `文件:行` | …… | …… |
| 重要 | `文件:行` | …… | …… |
| 建议 | `文件:行` | …… | …… |

（无问题则写"逐项检查未发现阻断/重要/建议级问题，质量过关"。）

### 结论
<无阻断/重要则一句话带过可提交；有阻断/重要则说明停下来等用户决定：先修改还是照常提交>
```

报告里的问题要给得具体、可落地——指到文件和行、说清为什么是问题、给出怎么改的建议，让用户能直接据此决定或动手。问题不多时如实少列，不为凑数硬编问题。

### 步骤 3：确定提交信息格式

提交信息要贴合项目惯例，按下面的优先级确定格式，**正文一律用中文**（技术术语、代码标识符保持原文）：

1. **先查项目规则。** 读 `.ctxloom/rules/other.md`（以及同目录其他规则文件），看有没有约定统一的提交信息格式。有就严格照它来。
2. **没有则跟随仓库历史。** 跑 `git log --oneline -20` 看最近的提交长什么样，归纳出仓库的惯用风格并沿用。仓库历史可能是 `feat(scope): xxx` 这种 Conventional Commits 风格，也可能是"功能：xxx""修复：xxx"这种中文前缀风格，或两者混合。挑其中占主流、且与本次改动性质相符的一种沿用。**正文语言跟随仓库历史的主流语言**(历史多为中文就用中文，多为英文就用英文)，技术术语、代码标识符保持原文。

举例（跟随仓库历史风格）：

**示例 1（新功能）：**
改动内容：实现了某功能，并归档了对应的需求与设计文档
提交信息：`feat(scope): 实现XX功能` （或 `功能：实现XX功能`，取仓库主流风格其一）

**示例 2（修复）：**
改动内容：修复了某处取数/渲染错误
提交信息：`fix(scope): 修复XX逻辑`

提交信息要让人一眼看出这次做了什么。本次提交把代码改动与文档归档合并为一个提交，提交信息以**主体的代码改动**为主来描述，文档归档可在正文补一句带过。

### 步骤 4：提炼归档本次功能的过程文档

本次功能已开发完、验证通过，把它的过程沉淀成一份归档成品。**归档是提炼，不是移动也不是复制**（见硬性规则）：读本功能在 `.ctxloom/` 下的活文档（`requirements/<功能>.md`、`designs/<功能>.md`、`tasks/<功能>.md`），把它们**提炼归纳总结**成一份精炼的 `.ctxloom/archive/<功能>.md`。

- 归档成品写什么：这个功能**最终做成了什么**——核心需求一句话、最终采用的技术方案与关键决策、与最初设想相比的偏差/调整、主要改动点、遗留项或后续注意事项。它是给"以后想快速回看这个功能"的人读的浓缩结论，不是过程流水账。
- frontmatter 与关联链接：`archive/<功能>.md` 头部按 `0-init-project-context`《Frontmatter 规范》写好 `type: Archive`、`feature: <功能>`、`status: delivered`、`title`、`timestamp`（交付时间，精确到秒 `YYYY-MM-DD HH:MM:SS`）；正文开头加关联指引行（`> 关联：需求 [/requirements/<功能>.md] ｜ 设计 [/designs/<功能>.md] ｜ 执行 [/tasks/<功能>.md] ｜ 归档 [/archive/<功能>.md]`，见《交叉链接约定》），让归档与源活文档双向可达。
- 翻转源活文档状态：把本功能在 `requirements/`、`designs/`、`tasks/` 下的活文档 frontmatter 的 `status` 统一改为 `delivered`——这是它们随功能交付而"封存"的标记（活文档内容不动、不删不移，仅翻状态）。缺对应活文档则跳过其状态翻转。
- 活文档怎么处理：`requirements/`、`designs/`、`tasks/` 下的活文档**留在原处**作详细过程记录，不删除、不移动——归档区是提炼后的成品，活文档是完整过程，两者并存、各司其职。
- 顺手更新索引：在 `.ctxloom/archive/README.md` 的已交付索引里登记本功能，索引用 OKF index 风格的链接列表（`* [标题](/archive/<功能>.md) - 一句话摘要，<日期> 交付`，见 `0-init-project-context`《各 README.md 索引的内容》）。同时建议把本功能从 `.ctxloom/requirements/README.md` 的「在做/计划中」索引移除或标记已交付，避免两份索引重复。
- 若 `.ctxloom/archive/` 目录不存在，先 `mkdir -p` 创建。若本功能在 `.ctxloom/` 下找不到对应活文档（如跳过了前面阶段），据现有信息尽量提炼，或向用户说明缺哪份、归档相应从简。

提炼出的 `archive/<功能>.md` 与索引更新，作为本次提交的一部分一起落库。

### 步骤 5：暂存、确认、提交

把改动暂存好，连同提交信息一起摆给用户确认，确认通过再提交：

- **暂存**：把本次相关的代码改动和归档文档变更加入暂存区。优先按文件精确 `git add`，避免 `git add .` 把无关改动（如 `.claude/settings.local.json`、`.omc/` 状态文件等）顺带提交进去。
- **扫敏感文件**：暂存后再看一眼暂存清单（`git status`），按硬性规则确认没有误入敏感文件。
- **交用户确认**：把"质量检查结论 + 将提交的文件清单 + 拟用的提交信息"清楚地呈现给用户，请其过目。这是提交前最后一道关。若步骤 2 存在阻断/重要问题，此处必须已就"先修改 / 照常提交"得到用户的明确选择，不带着未决的阻断/重要问题径直提交。
- **提交**：用户确认后执行 `git commit`。提交信息按步骤 3 的格式。不擅自加 `--amend`、不加 `--no-verify`（除非用户明确要求跳过 hook）。

### 步骤 6：询问是否推送远端

提交完成后，把提交结果简要告诉用户（提交了哪个 commit、在哪个分支），然后**询问用户是否要 push 到远端**：

- 用户同意 → 执行 push。若是新分支，用 `git push -u origin <分支名>` 建立跟踪。
- 用户不要 / 没明确表态 → 停在本地提交这一步，告知用户改动已在本地提交、可自行决定何时推送。

不要在用户没点头的情况下擅自 push。

### 步骤 7：更新索引，提示复盘

提交完成后：

1. 到 `.ctxloom/tasks/current.md` 把本功能标为已交付/移出在途（该功能 0–6 流程至此走完）；若 `current.md` 还有其他在途功能则保留它们（`current.md` 缺失则跳过）。
2. 确认 `.ctxloom/archive/README.md` 的已交付索引已登记本功能、含交付日期（步骤 4 已登记则核对即可；archive 缺失则跳过）。这一步让本功能进入历史长期记忆。
3. **顺势提示复盘**：建议用户接着用 `6-retrospective` 做一次复盘，把本次开发踩的坑与定的约定沉淀进 `learnings.md`/项目规则，让流水线越用越聪明。只是提示，由用户决定是否进行。

记住硬性规则：提交前让用户过目、提交前做质量清单检查（有阻断/重要停下来让用户选）、不直接提交主干、敏感文件不入库、push 先问、归档是提炼不是复制。语言相关的检查项一律读项目规则，不写死。
