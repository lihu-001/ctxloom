# 提交审查

本技能承接 4-executing-plan 完成的代码，做提交前的最后把关：差异审查、安全检查、生成提交信息、更新活文档状态。它的价值在于**用一双新眼睛检查改动，避免带着问题入库**。

## 简繁自适应（按基线档缩放）

本技能支持按任务复杂度缩放深浅。进入时**先读任务活文档 `.ctxloom/tasks/<功能>.md` 的基线档**：

- **标准档**：改动面小时，主智能体直接审查差异、做安全检查、生成提交信息。
- **深度档**：改动面大时（多文件、跨模块），派 `diff-reviewer` 子智能体做独立审查，输出问题清单。

**硬地板（标准档也不能省）**：① **安全检查不可跳过**，必须按 `.ctxloom/rules/security.md` 过一遍；② **提交信息必须规范**，读 `.ctxloom/rules/other.md` 的提交格式约定。

## 硬性规则

- **只审查差异，不改代码。** 本技能的职责是把关，不是返工。发现问题时，列出问题清单交给用户决定——是回到执行阶段修复、还是接受风险提交。
- **安全检查必须做。** 按 `.ctxloom/rules/security.md` 固化的本项目高危模式逐项核对差异。
- **提交信息读项目规则。** 格式、前缀、scope 约定读 `.ctxloom/rules/other.md`。
- **不替用户执行 commit/push。** 只生成提交信息并提示用户自行提交。

## 工作流程

### 步骤 1：获取差异

用 `git diff` 获取当前分支相对于主分支的差异。差异大时（超过 1000 行或 20 个文件），派 `diff-reviewer` 子智能体审查（协议见 `agents/diff-reviewer/AGENT.md`）。

### 步骤 2：质量审查

审查差异，按以下维度检查：

- **代码质量**：明显的逻辑错误、资源泄露、异常处理缺失
- **项目规则遵循**：命名、分层、事务、审计字段等是否符合 `.ctxloom/rules/`
- **安全检查（必做）**：按 `.ctxloom/rules/security.md` 的高危模式逐项核对
- **测试覆盖**：关键逻辑是否有测试

发现问题时，列出问题清单（阻断/重要/建议三档）交给用户。

### 步骤 3：生成提交信息

读取 `.ctxloom/rules/other.md` 的提交格式约定，生成规范的提交信息。格式通常包含：
- type（feat/fix/refactor/docs 等）
- scope（可选）
- subject（简短描述）
- body（详细说明，可选）
- footer（关联 issue，可选）

### 步骤 4：更新活文档状态

- 将需求/设计/任务活文档的 `status` 改为 `delivered`
- 从 `.ctxloom/tasks/current.md` 移除本功能
- 提示用户：代码已通过审查，可自行提交并进入 `ctxloom:6-retrospective` 复盘
