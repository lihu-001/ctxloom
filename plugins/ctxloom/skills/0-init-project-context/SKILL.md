## 概述

本技能为项目建立一套标准化的「项目上下文」目录，让后续的 AI 协作（需求分析、设计、编码、评审）有统一的背景知识和规则可依据。

核心产物是项目根目录下的 `.ctxloom/` 目录，它把几类信息固化下来：

- **项目背景与架构** —— 这个项目是做什么的、怎么分层、关键模块如何协作（落点 `project.md`）
- **项目规则** —— 代码风格、编码约定等团队规则，必须是**通用规则**而非针对某个文件的描述；其中**构建命令**、**测试方式**、**安全高危点**三类信息是下游技能(计划/执行/提交)落地验证与安全检查的强依赖，本技能要专门固化下来（分别落 `rules/build.md`、`rules/testing.md`、`rules/security.md`）
- **长期记忆** —— `learnings.md`(开发中踩的坑与定的约定的沉淀)。本项目历史需求的索引不再单设台账，而是由 `archive/`（已交付功能的提炼归档）+ 各 `requirements/README.md` 自然承载
- **过程文档（活文档）与归档区** —— 每个功能一份贯穿需求/设计/执行的活文档（`requirements/`、`designs/`、`tasks/`），交付后提炼归纳成 `archive/<功能>.md`

> **语言无关原则。** 这套 0–6 技能要跨 Java、Vue 等多种项目复用，因此技能定义本身只写语言无关的方法论；一切与语言/项目相关的具体内容(构建命令、测试框架与命令、安全高危模式、提交规范)都固化在本项目的 `.ctxloom/rules/` 里，由下游技能按需读取。这就是为什么本技能必须把「构建命令」「测试方式」「安全高危点」固化清楚——下游技能不写死 `mvn`/`npm`，而是来这里读。

之所以要先建立这套上下文，是因为它一次提炼、长期复用：后续每次开发都能直接读取这些文件，不必重新摸索项目，团队成员之间也有一致的规则基线。

## 硬性规则

- 规则文件里写的规则**必须通用**。规则描述的是”在这个项目里应当怎么做”，适用于未来任意新代码，而不是”某个类现在是怎么写的”。这是本技能最重要的约束——一条只对单个文件成立的描述不是规则。
- 不臆造项目背景与架构。只写从代码、配置、文档中**实际检索到**的信息；无法确认的部分明确标注「待补充」，不要编造。
- 已存在的内容默认保留并补全/刷新，不要无差别覆盖用户已经手写的内容（详见「老项目：补全与刷新」）。
- 项目背景/架构以 `.ctxloom/project.md` 为**权威来源**。项目若已有 `CLAUDE.md`，把它当作检索信号源即可，不要把长篇背景/架构同时塞进 `CLAUDE.md` 与本目录而形成两套并存、各自漂移的描述（详见「与 CLAUDE.md 的关系」）。
- **每份文档自描述身份、消费时宽容。** `.ctxloom/` 是一个自描述的知识目录：每个 `.md`（`README.md` 除外）头部带一段 YAML frontmatter 标明它的 `type` 等元信息（见「Frontmatter 规范」）。但**读取**这些文档时必须**宽容**——容忍缺 frontmatter、缺字段、未知 `type`、坏的交叉链接，绝不因格式不全而报错或拒绝处理。这条对本技能与下游 1–6 同样适用：frontmatter 是叠加增强，缺了也要能照常按路径读正文。这把本目录一贯的「待补充」理念正式化为消费契约。

## 上下文管理（执行流程通用）

第 2、3 步的项目检索会积累大量上下文（读取文件、调用 codegraph、分析代码）。为避免自动压缩导致已完成工作丢失，**必须在关键节点阶段性落盘**：

### 落盘检查点与策略

1. **强制落盘节点**（每完成一个阶段立即落盘，不等到全部完成）：
   - ✅ **完成第 1 步后**：如已创建目录结构与空文件，立即落盘（Write 所有已建的文件）
   - ✅ **完成第 2 步后**：`project.md` 填充完毕，立即写入磁盘
   - ✅ **完成第 3 步后**：所有 `rules/*.md` 提炼完毕，立即全部写入磁盘
   - ✅ **完成第 4 步后**：`learnings.md`、`current.md`、各 `README.md` 全部写入磁盘

2. **渐进式写入策略**（特大项目）：
   - 对于超大项目（100+ 模块、1000+ 文件、或明显感受到上下文压力），采用”分批检索、即时落盘”：
     - 先检索核心模块与技术栈，写入 `project.md` 的**初版**（含「待补充」标记）
     - 再检索规则相关代码样本，补充 `project.md` 并**立即覆盖写入**
     - 然后提炼规则文件，每完成 2-3 个规则文件就**批量写入**
     - 最后补充细节，而不是等所有分析完才开始写
   - 这样即使中途触发压缩，已落盘的内容不会丢失，后续可从磁盘读回继续补充

3. **落盘后的引用原则**：
   - 落盘后，如需回顾已完成阶段的内容，**从磁盘读取**而不是依赖对话历史
   - 例如：第 3 步提炼规则时需要参考架构信息，应 Read `project.md`，而不是回溯对话中的分析结果

4. **上下文压力信号**（满足任一条件立即落盘当前已完成的内容）：
   - 已调用 codegraph/Read 工具超过 30 次
   - 已读取的文件内容明显很长（单个文件 > 500 行，或累计读取 > 3000 行）
   - 已在某一步骤中分析/推理超过 20 秒
   - 感受到对话历史已经很长

5. **老项目刷新时的落盘**：
   - 刷新模式同样适用上述规则
   - 每刷新完一个文件（如 `project.md`）立即写回，不等所有文件都刷新完

### 为什么必须这样做

- **自动压缩的不可预测性**：压缩可能在任意时刻触发，压缩后的摘要可能丢失检索过程中的细节
- **检索成本高**：重新检索项目比从磁盘读取已落盘的文件成本高得多
- **用户体验**：如果因压缩导致某个阶段需要重做，用户体验极差
- **渐进式交付**：阶段性落盘让用户能看到进度，即使中途中断也有部分成果

## 目录结构

`.ctxloom/` 的标准结构如下。**带 `.md` 后缀的是文件，其余都是文件夹**。除 `README.md` 外，每个 `.md` 文件头部都带一段 YAML frontmatter 标明身份（见「Frontmatter 规范」）：

```
.ctxloom/
    README.md                        # 文件：本目录的导览与使用约定（含 Frontmatter 规范，唯一可带 okf_version 的文件）
    log.md                           # 文件：上下文目录的演进日志（可选，记结构与架构级变更，最新在前）
    project.md                       # 文件：项目背景、技术栈、整体架构（架构权威来源）
    learnings.md                     # 文件：踩坑经验与临时约定的沉淀（经验回流，6 追加）
    pending.md                       # 文件：待确认问题的暂存区（开放问题，确认后清理）
    rules/                           # 文件夹：编码前参考的规则基线
        README.md                    # 文件：本目录用途说明 + 各规则文件导引
        code-style.md                # 文件：命名 / 格式 / 注释等外观风格
        coding.md                    # 文件：分层职责 / 异常 / 日志 / 事务 / DTO 转换等工程约定
        build.md                     # 文件：构建命令（下游 3/4/5 强依赖）
        testing.md                   # 文件：测试框架与命令、难测部分的手动验证（下游 3/4/5 强依赖）
        security.md                  # 文件：本项目技术栈的安全高危点（下游 2/5 强依赖）
        other.md                     # 文件：其他项目特有约定（提交格式、命名前缀等）
    requirements/                    # 文件夹：各功能的需求活文档
        README.md                    # 文件：本目录用途说明 + 功能需求索引（OKF index 风格）
        <功能>.md                    # 文件：某功能的需求活文档（一功能一份，无日期戳）
    designs/                         # 文件夹：各功能的技术设计活文档
        README.md                    # 文件：本目录用途说明 + 索引
        <功能>.md                   # 文件：某功能的技术设计活文档
    tasks/                           # 文件夹：执行阶段的任务活文档
        README.md                    # 文件：本目录用途说明
        current.md                   # 文件：当前在做什么、走到哪一步、下一步是什么
        <功能>.md                    # 文件：某功能的实现计划与执行进度活文档
    archive/                         # 文件夹：已交付功能的提炼归档
        README.md                    # 文件：本目录用途说明 + 已交付功能索引（OKF index 风格）
        <功能>.md                    # 文件：某功能交付后提炼归纳的成品（兼历史索引）
```

`requirements/`、`designs/`、`tasks/`、`archive/` 下的 `<功能>.md` 是按功能纵切的**活文档**：同一个功能从需求到设计到执行用**同名文件**贯穿（如 `requirements/coupon.md` → `designs/coupon.md` → `tasks/coupon.md` → 交付后 `archive/coupon.md`），顺着同名文件就能看清一个功能的全貌。本技能只负责把这些文件夹和各自的 `README.md`、以及 `current.md` 建好，不预放功能文件。

> **活文档模型（取代日期戳快照）。** 过程文档不再用 `<标题>-需求分析-yyyyMMdd.md` 这种带日期戳的历史快照，而是一个功能一份**活文档**，始终反映该功能的当前事实。澄清/设计/执行过程中的决策**直接写进活文档**（不再单开 `-草稿-yyyyMMdd` 文件）；多数时候团队要的是最新版，不是版本堆。需要回看历史时，靠 git 历史，而非堆文件。

> **不再设阶段状态表与需求台账。** 「当前哪个功能走到哪一步」由 `tasks/current.md` 承载（轻量记录当前功能/阶段/下一步），不再用 `流水线状态.md` 跟踪 0–6 的细粒度阶段。「本项目历来做过哪些功能」由 `archive/`（已交付）+ `requirements/README.md`（在做/计划中）的索引自然承载，不再单设 `需求台账.md`。

> **归档 = 提炼，不是复制。** 功能交付后，由 `5-committing-changes` 把该功能的需求/设计/执行活文档**提炼归纳总结**成一份精炼的 `archive/<功能>.md` 成品——它是"这个功能最终做成了什么"的浓缩结论，不是把活文档原样搬过去。活文档本身留在原处作详细过程记录。

> **目录契约的单一来源。** 上面这棵目录树是整套 0–6 技能流程的**权威定义**。后续技能（`1-requirements-analysis` 到 `6-retrospective`）在落盘、归档、读取规则与记忆时会引用其中的具体路径（如 `requirements/<功能>.md`、`rules/build.md`、`rules/security.md`、`learnings.md`、`archive/<功能>.md`）。若要调整结构（重命名或移动目录），**以这里为准**，并同步更新引用这些路径的技能，避免契约在多处漂移。

## Frontmatter 规范

`.ctxloom/` 是一个**自描述的知识目录**：除 `README.md` 外，每个 `.md` 文件头部都带一段 YAML frontmatter，让人和 AI 不必读技能定义就知道这份文档是什么、属于哪个功能、处于什么阶段。本节是这套规范的**权威定义**——下游 1–6 写文档时按此填写，本技能新建/刷新文档时也按此填写，并把同一份规范写进 `.ctxloom/README.md` 供仓库浏览者就地查阅。

**这套规范是叠加增强，不是门禁。** 配合「硬性规则」里的宽容消费条款：写文档时按规范填，但读文档时容忍缺字段、未知 `type`、坏链——绝不因 frontmatter 不全而拒绝处理。

### 字段定义

frontmatter 用 `---` 成对包裹，置于文件最顶部。字段分三档：

**全局必需（唯一硬要求）：**

- `type` —— 标明这份文档的种类，供路由与过滤。取值见下表，**不集中注册**，未知取值按"通用文档"宽容处理。

**全局推荐：**

- `title` —— 人类可读标题。缺省时可由文件名推导。
- `timestamp` —— 最后一次有意义改动的时间，精确到秒，用年月日时分秒格式 `YYYY-MM-DD HH:MM:SS`（如 `2026-06-22 14:30:05`）。

**按 `type` 的专属字段（仅在解决实际需要时填，不强求）：**

| 文件 | `type` | 专属字段 | 含义 |
|---|---|---|---|
| `project.md` | `Project` | —— | 项目背景与架构权威源 |
| `rules/*.md` | `Rule` | `downstream`（可选，如 `[3, 4, 5]`） | 标明此规则是下游哪些技能的强依赖，便于精确定位 |
| `learnings.md` | `Learnings` | —— | 经验沉淀区 |
| `pending.md` | `Pending` | —— | 待确认问题暂存 |
| `requirements/<功能>.md` | `Requirement` | `feature`、`status`、`resource` | 见下 |
| `designs/<功能>.md` | `Design` | `feature`、`status` | 见下 |
| `tasks/<功能>.md` | `Task` | `feature`、`status`、`branch` | 见下 |
| `archive/<功能>.md` | `Archive` | `feature`、`status` | 见下 |

活文档专属字段的含义：

- `feature` —— 该文档所属功能的简短英文/拼音名（如 `coupon`），与文件名一致。它把原本只靠"文件同名"的隐式串联变成**可被聚合查询的显式关系**——按 `feature` 就能把一个功能的 需求/设计/任务/归档 全部捞出来。
- `status` —— 该功能当前所处阶段，取值：`draft`（草稿）｜`in-progress`（进行中）｜`reviewing`（评审中）｜`delivered`（已交付）。让消费方一眼分清在做/已交付，不必去翻 README 索引表。
- `resource` —— 仅 `Requirement` 用：指向 PRD / 原型的**链接或相对路径**（只存引用、不搬原件，呼应「提炼不是复制」）。无外部材料时省略。
- `branch` —— 仅 `Task` 用：执行阶段关联的 git 分支名（如 `feat/coupon`）。计划阶段可留占位、执行阶段(4)回填。

### 示例

需求活文档（绑定 PRD 资源）：

```yaml
---
type: Requirement
title: 优惠券发放与核销
feature: coupon
status: in-progress
resource: https://lanhuapp.com/web/#/item/project/...   # 只存引用
timestamp: 2026-06-22 14:30:05
---
```

规则文件（标注下游强依赖）：

```yaml
---
type: Rule
title: 构建命令
downstream: [3, 4, 5]
---
```

`.ctxloom/README.md` 是唯一可声明 bundle 版本的文件，其 frontmatter 可含 `okf_version: "0.1"`（其余 `README.md` 不带 frontmatter）。

## 交叉链接约定

活文档之间用标准 Markdown 链接显式互联，把"顺着一个功能看全貌"从**人脑记同名约定**变成**可点击、可遍历**的关系图。

**用 bundle 相对链接（推荐）：** 以 `/` 开头、相对 `.ctxloom/` 根解析，文档在子目录间移动时仍稳定。

每份活文档（`requirements/`、`designs/`、`tasks/`、`archive/` 下的 `<功能>.md`）正文**开头**放一行关联指引，指向该功能的上下游同名文档：

```markdown
> 关联：需求 [/requirements/coupon.md] ｜ 设计 [/designs/coupon.md] ｜ 执行 [/tasks/coupon.md] ｜ 归档 [/archive/coupon.md]
```

**坏链是合法的。** 功能尚未走到归档阶段时，`archive/coupon.md` 还不存在，那条链接就是坏的——这表示"尚未写的知识"，不是错误。消费方必须容忍坏链（见硬性规则的宽容消费）。



### 第 1 步：检测上下文目录是否存在

检查项目根目录下是否已有 `.ctxloom/`。

- **不存在** → 走「新建」分支：按上面的结构创建全部目录与文件（每个 `.md` 按「Frontmatter 规范」写好头部，活文档目录此时只建 `README.md`/`current.md`、不预放功能文件，故暂无需关联链接行），然后把目录与文件清单写入 `README.md`（见「README.md 的内容」）。`log.md` 可选——建议新建时一并创建并写入第一条 `Initialization` 记录（见「log.md 的内容」）。
- **已存在** → 走「补全与刷新」分支（见「老项目：补全与刷新」）：逐项比对结构，补建缺失的目录/文件，并刷新 `project.md` 与规则文件的内容。

无论哪个分支，最终都要保证目录结构完整、`README.md`/`project.md`/规则文件都有内容。

### 第 2 步：检索项目上下文，填充「project.md」

**先做检索方式预检（CodeGraph 优先）。** 本步要大量摸代码——技术栈、分层、模块协作、调用关系。结构化检索远比 grep/逐文件读高效且省上下文，优先走 CodeGraph：

1. **探测是否可用**：先尝试调用一次 CodeGraph（如 `codegraph_status`，或对本项目主目标的 `codegraph_context`）。
   - **能正常返回** → 本步与第 3 步的结构性检索一律优先用 CodeGraph：`codegraph_context` 摸某一主题/模块的全貌、`codegraph_files` 看目录与文件结构、`codegraph_search`/`codegraph_node` 定位与查看符号、`codegraph_impact`/`codegraph_callers` 看关键符号的波及面与调用方。grep/逐文件读只用于查字面文本（注释、配置字符串）或已定位文件后的细看。
   - **报「未初始化 / 工具不可用」** → 进入第 2 点。
2. **缺失时按缺哪层处理，必须征询用户、绝不静默安装**：
   - **项目无 `.codegraph/` 索引**（最常见）→ 提示用户：「这个项目还没建 CodeGraph 索引，建好后本次以及后续需求/设计/计划阶段的代码检索都会更快更准、更省上下文。要我建吗？（你也可自行执行 `! codegraph init -i`）」。建索引在大项目上耗时数分钟、占用磁盘，属改变项目状态的动作，**得到用户同意再建**；不同意 → 跳到第 3 点兜底。
   - **连 `codegraph` 命令都没有** → 只提示一句安装方式（全局安装 codegraph CLI），由用户自行决定，**不要静默 `npm install -g`**——那是改动用户机器，超出本技能职责。
3. **兜底不阻塞**：用户不启用 CodeGraph 时，用 grep/逐文件读照常完成检索与本技能的全部产出；并在第 4 步汇报里注明「本次未启用 CodeGraph，建议后续初始化以提升各阶段检索效率」。

确定检索方式后，检索项目以理解它的全貌。优先看这些信号源：

- 构建/依赖文件（如 `pom.xml`、`package.json`、`build.gradle`、`Cargo.toml`、`go.mod`、`requirements.txt`）——判断技术栈、模块划分、关键依赖
- 根目录的 `README`、已有的 `CLAUDE.md` 或文档目录——项目意图与背景
- 目录结构与分层（哪些是入口、领域层、数据层、任务/调度、对外接口）
- 配置文件（数据库、缓存、消息队列、外部服务地址）——判断依赖的外部系统

把检索结果组织进 `project.md`，建议包含：

```markdown
# 项目背景与架构

## 项目背景
（这个项目解决什么问题、面向什么业务/用户。检索不到时标注「待补充」）

## 技术栈
（语言、框架、构建工具、关键依赖及其用途）

## 模块与目录结构
（顶层模块/目录各自的职责，多模块项目尤其要说清模块间关系）

## 整体架构
（分层方式、核心数据流、关键组件如何协作；可用简单的文字或列表描述）

## 外部依赖
（数据库、缓存、消息队列、第三方服务等，及连接/集成方式）
```

只填能从项目中确认的内容，确认不了的写「待补充」。

### 第 3 步：检索项目上下文，提炼项目规则

从项目现有代码中**归纳**出通用规则，写入 `rules/` 下对应文件。归纳的方法：观察多个文件中**反复出现**的一致做法，把它抽象成一条对未来代码同样适用的规则。

- `code-style.md` —— 命名约定（类/方法/变量/常量/包）、缩进与格式、注释风格、文件组织等**外观**层面的约定。
- `coding.md` —— 分层与职责边界、异常处理方式、日志规范、参数校验、事务使用、DTO/VO/Entity 的转换约定、依赖注入方式、接口返回结构等**工程实践**约定。（数据库访问、对外接口等约定起步并入此处，内容多了再按需拆分独立文件。）
- `build.md` —— **下游技能(3/4/5)的强依赖，必须固化清楚**。本项目怎么编译/打包（如 `mvn -pl xxx compile`、`npm run build`、`cargo build`、`go build ./...`），多模块项目说清各模块怎么单独构建。语言无关的下游技能不写死命令，而是来这里读。
- `testing.md` —— **下游技能(3/4/5)的强依赖**。用什么测试框架、怎么跑全量测试、怎么跑单个测试类/文件（如 `mvn -Dtest=XxxTest test`、`npm run test -- xxx.spec.ts`、`pytest path::case`）；项目测试薄弱或没有测试时，如实写明，并给出"难单测部分怎么手动验证"的替代手段（起服务调接口、跑 Job 核对落库等）。
- `security.md` —— **下游技能(2/5)安全检查的强依赖**。本技术栈在本项目里的真实高危模式，供设计评审与提交质检引用。如 Java：字符串拼 SQL、MyBatis `${}` 直插、硬编码密钥/地址、敏感信息进日志；Vue/前端：`v-html` 导致 XSS、密钥/token 打进前端包、`dangerouslySetInnerHTML`、未校验的 URL 跳转等。按本项目实际技术栈写。
- `other.md` —— 不属于上面几类、但项目里确实存在的特殊约定（如特定的注解用法、命名前缀、目录约定、提交信息格式等）。检索不到此类约定时，放一行说明「暂无项目特有的自定义规则，后续按需补充」即可。

**关键要求——规则必须通用：**

每条规则都应当能直接套用到一段还没写的新代码上。写规则时做一次自检：「如果明天新增一个类，这条规则能指导它怎么写吗？」能，才是规则。

```markdown
# 反例（不要这样写——这是对现状的描述，不是规则）
- UserServiceImpl 里用了 @Transactional

# 正例（通用规则）
- Service 实现类中涉及多次写库的方法统一加 @Transactional，保证原子性
```

**当项目里检索不到足够依据时**：用该技术栈/语言社区公认的**通用最佳实践**兜底填充（例如 Java 项目可参考阿里巴巴 Java 开发手册的通用条目、Spring 项目的常见分层约定；Vue 项目可参考官方风格指南）。兜底的规则同样必须通用，并在文件开头注明「以下规则部分基于通用最佳实践，请团队结合实际情况校准」，方便用户区分哪些是从项目里提炼的、哪些是补充的。

### 第 4 步：建立长期记忆文件

建好 `learnings.md`（经验回流的沉淀区）。它区别于 `tasks/current.md`：`current.md` 只管当前在途功能走到哪一步，而 `learnings.md` 是**跨功能、长期累积**的记忆。

- **`learnings.md`** —— 经验回流的沉淀区，由 `6-retrospective` 在每次复盘时追加。本技能只负责建好空文件（含 `type: Learnings` frontmatter 与表头说明），不预填条目。内容结构见「learnings.md 的内容」。

> **历史需求索引不再单设台账。** 「本项目历来做过哪些功能」由 `archive/`（已交付功能的提炼归档，附 `archive/README.md` 索引）+ `requirements/README.md`（在做/计划中的功能索引）自然承载。本技能新建时把这些 `README.md` 建好（含表头说明、暂不放数据行）；老项目刷新时，尽量从已有过程文档回填索引行。

### 第 5 步：完成后告知用户

简要汇报：建立/补全了哪些目录与文件、背景架构提炼了哪些要点、规则文件各写入了几条规则、各文档是否已按规范带上 frontmatter、哪些地方标了「待补充」或基于通用最佳实践需要团队校准。

## 老项目：补全与刷新

当 `.ctxloom/` 已存在时（典型是老项目第二次运行本技能）：

1. **结构补全**：逐项对照「目录结构」，缺失的目录和文件补建出来；已存在的目录/文件保留。
2. **内容刷新**：重新检索项目，刷新 `project.md` 与规则文件——但**不要无差别覆盖用户手写的内容**。做法是：
   - 若文件内容明显是本技能上次生成的（结构、措辞一致），可整体重写为最新版本。
   - 若文件中包含用户后来补充/修改的内容（手写规则、补的背景说明），保留这些内容，只在其基础上补充新发现的信息，必要时用「（自动补充）」之类的标注区分，避免覆盖用户的劳动成果。
3. `README.md` 中的清单同步更新为当前实际结构。
4. **回填历史索引**：扫描 `archive/`、`requirements/`、`designs/`、`tasks/` 下已有的活文档与归档文档，把 `archive/README.md`（已交付）与 `requirements/README.md`（在做/计划中）的索引补齐，使其与实际文件吻合。
5. **补建长期记忆文件**：若缺 `learnings.md`，新建空文件（含表头）。已存在则保留用户内容，不覆盖。
6. **补建规则强依赖文件**：老项目若缺 `rules/build.md`、`rules/testing.md`、`rules/security.md`，按第 3 步的要求新建并固化构建命令、测试方式、安全高危点——这是下游技能跨语言运行的强依赖，不能缺。
7. **从旧结构迁移（如适用）**：若项目残留早期版本的 `.项目上下文/` 目录（含 `项目背景与架构.md`、`项目规则/构建与验证.md`、`流水线状态.md`、`需求台账.md`、临时/归档分层目录等），提示用户这是旧结构，建议迁移到新结构（`project.md`、拆分后的 `rules/`、活文档与 `archive/`），迁移涉及移动/合并用户内容，**得到用户同意再做**，不静默改动。
8. **升级 frontmatter 与交叉链接**：老项目的既有 `.md` 若缺「Frontmatter 规范」要求的头部（早期版本无 frontmatter），按规范**补写**——`type` 必补，`feature`/`status` 等按文件类型尽量从内容/文件名推断填上；活文档若缺开头的关联指引行，按「交叉链接约定」补一行。这是纯叠加，不动正文。把 `README.md` 升级为含「Frontmatter 规范」「交叉链接约定」两节的内置副本（见「README.md 的内容」），各索引类 `README.md` 的表格若仍是旧式表格，改写为 OKF index 风格的链接列表。判断不准是否用户手写时，宁可只补 frontmatter、不动正文。
9. **补建演进日志**：若缺 `log.md`，可补建并在顶部写一条本次刷新记录（如 `**Refactor**: 升级为 OKF bundle，补齐 frontmatter 与交叉链接`）；已存在则在顶部追加。`log.md` 是可选项，用户不需要时跳过。

判断不准时，宁可保留用户内容、把新信息追加上去，也不要直接抹掉。

## 与 CLAUDE.md 的关系

项目背景、架构、技术栈这些信息，在 `CLAUDE.md` 与 `.ctxloom/project.md` 里容易各写一份、日久漂移。两者分工应当明确：

- **`.ctxloom/project.md` 是权威、详尽的来源。** 它按需读取（0–6 技能与人都从这里了解项目），适合放完整的背景与架构。
- **`CLAUDE.md` 保持精简、只留指针。** 它会被每个会话自动载入、常驻上下文，长篇背景放这里既费 token 又与本目录重复。它该承载的是「每个会话都必须立刻知道」的少量要点（构建/测试命令、必须遵守的硬约束、以及一句指向 `.ctxloom/` 的指引），而不是完整架构叙述。

据此，本技能的处理：

1. **检索阶段**：把已有的 `CLAUDE.md` 当作信号源读取，但项目背景/架构的最终落点是 `project.md`，不把同样的长篇内容再灌进 `CLAUDE.md`。
2. **补指针（征得同意）**：若项目已有 `CLAUDE.md` 却没有指向 `.ctxloom/` 的指引，建议补一行短指针（如「项目背景/架构/规则见 `.ctxloom/`；开发按 0–6 技能流程」）——这会改动用户的 `CLAUDE.md`，**先征得用户同意再加**。项目若没有 `CLAUDE.md`，本技能不替它新建（要不要 `CLAUDE.md` 是另一回事，可用原生 `/init`）。
3. **刷新阶段对账**：老项目刷新时，若发现 `CLAUDE.md` 里的背景/架构与 `project.md` 不一致，不要默默以某一方覆盖另一方，把差异点摆给用户，由用户决定以哪份为准、要不要把 `CLAUDE.md` 里的长背景收敛成指针。

## learnings.md 的内容

`learnings.md` 是**经验回流的沉淀区**：开发中踩的坑、临时定的约定，由 `6-retrospective` 在每次复盘时追加。够通用、能指导未来代码的，会被提炼进 `rules/`；其余不够格成为规则的，留在此处作为教训记录。本技能只负责建好空文件(含表头说明)，不预填条目。

```markdown
---
type: Learnings
title: 经验沉淀
---
# 经验沉淀（learnings）

> 开发中踩的坑、定的约定。够通用的会被 6-retrospective 提炼进 rules/；其余留此作教训。

## <日期> <功能名>
- **坑**：<问题现象>
- **根因**：<为什么会这样>
- **解法/约定**：<以后怎么避免>
- **是否已入规则**：是(→ coding.md 第 X 条) / 否(仅记录)
```

## tasks/current.md 的内容

`current.md` 是当前在途工作的**轻量索引**：用几行记录「现在在做哪个功能、走到 0–6 的哪一步、下一步是什么」，让任一技能在任一会话都能快速定位当前上下文，而不必扫目录猜测。它只管**在途**，交付完成后该功能的长期记录归 `archive/<功能>.md`。

```markdown
---
type: Current
title: 当前在途工作
timestamp: <最后更新时间，YYYY-MM-DD HH:MM:SS>
---
# 当前工作

> 当前在途功能的轻量索引。1–5 技能推进到新阶段时顺手更新；交付后归档到 archive/。

- **功能**：coupon（优惠券发放）
- **当前阶段**：技术设计 → 实现计划
- **活文档**：[/requirements/coupon.md] ✓ ｜ [/designs/coupon.md] ✓ ｜ [/tasks/coupon.md]（进行中）
- **关联分支**：feat/coupon
- **下一步**：拆解实现步骤
```

- **新建项目**：创建本文件，写入表头与说明、暂不放数据行（尚无在途功能）。
- **老项目刷新**：扫描 `requirements/`、`designs/`、`tasks/` 下的活文档，推断当前在途功能与阶段，填入本文件。

## 各 README.md 的内容

`rules/README.md`、`requirements/README.md`、`designs/README.md`、`tasks/README.md`、`archive/README.md` 各自说明本目录用途。其中 `requirements/README.md` 与 `archive/README.md` 还**兼当功能索引**（取代旧的需求台账）；`rules/README.md` 不是索引，而是给出本目录各规则文件的**导引**（每个规则文件是干什么的、哪些是下游强依赖）：

- `rules/README.md` —— 说明本目录用途，并导引各规则文件（含哪些是下游强依赖）。它**不随项目功能增减而变动**，新建/刷新时按下面的范例固定写入即可。
- `requirements/README.md` —— 索引**在做/计划中**的功能。
- `archive/README.md` —— 索引**已交付**的功能，是本项目历史需求的长期回看入口。

`rules/README.md` 范例（导引各规则文件，非索引；不带 frontmatter）：

```markdown
# 项目规则（rules）

> 编码前参考的规则基线。这里写的是"在这个项目里应当怎么做"，每条规则都应能直接套用到一段还没写的新代码上，而不是"某个类现在怎么写"。

## 各规则文件

* [code-style.md](/rules/code-style.md) - 命名 / 缩进格式 / 注释等外观风格
* [coding.md](/rules/coding.md) - 分层职责 / 异常 / 日志 / 校验 / 事务 / DTO 转换等工程实践约定
* [build.md](/rules/build.md) - 构建 / 打包命令（下游 3/4/5 技能强依赖）
* [testing.md](/rules/testing.md) - 测试框架与命令、难测部分的手动验证（下游 3/4/5 技能强依赖）
* [security.md](/rules/security.md) - 本项目技术栈的安全高危点（下游 2/5 技能强依赖）
* [other.md](/rules/other.md) - 其他项目特有约定（提交格式、命名前缀等）

发现新的团队约定时补充到对应文件；规则必须通用。
```

**索引采用 OKF index 风格的分组链接列表**（`* [标题](/bundle相对路径) - 描述`），而非表格——这样既能人读，又能被当作渐进式披露的 index 由 AI 消费，链接还可直接跳到对应活文档。`README.md` 本身不带 frontmatter。

```markdown
# 需求活文档（requirements）

> 每个功能一份需求活文档（<功能>.md），始终反映该功能需求的当前事实。

## 在做 / 计划中

* [优惠券发放与核销](/requirements/coupon.md) - 优惠券的发放与核销，设计中
* [登录优化](/requirements/login-opt.md) - 登录流程精简，需求分析中
```

```markdown
# 归档（archive）

> 已交付功能的提炼归档，本项目历史需求的长期回看入口。

## 已交付

* [优惠券发放与核销](/archive/coupon.md) - 优惠券发放与核销，2026-06-20 交付
```

条目描述取自对应活文档 frontmatter 的 `description`/`title`。新建项目时各 `README.md` 写入用途说明与分组标题、暂不放数据行；老项目刷新时尽量从已有活文档/归档回填索引（可读各活文档的 frontmatter 取 `title`/`status` 填充）。

## log.md 的内容

`log.md` 是 `.ctxloom/` 这套目录**自身的演进日志**（可选），按 OKF 日志格式记录——**只记目录结构层面与架构级的变更**（如目录升级、新增/重命名规则文件、架构大调整），功能级的细节仍由活文档与 git 历史承载，不往这里堆。它不带 frontmatter，日期标题用 `YYYY-MM-DD`、最新在前：

```markdown
# 上下文目录更新日志

## 2026-06-22
* **Refactor**: 各活文档补齐 frontmatter 与交叉链接。

## 2026-06-20
* **Initialization**: 建立 .ctxloom/ 初始目录结构。
```

leading 加粗词（`**Initialization**`、`**Update**`、`**Refactor**`、`**Deprecation**` 等）是约定、非强制。新建项目时创建本文件并写入第一条 `Initialization`；老项目刷新时若无则可补建，有则在顶部追加本次刷新记录。

## README.md 的内容

`.ctxloom/README.md` 是整个上下文目录的导览，也是这套 frontmatter / 链接规范在仓库里的**内置副本**——让任何打开这个目录的人或 AI，不查技能定义就知道这里的文档头部该怎么写。它需包含：

1. **目录与文件清单** —— 完整列出 `.ctxloom/` 下的目录与文件树（即「目录结构」那棵树），让人一眼看清结构。
2. **每个文件/目录的用途说明** —— 逐项说明它是干什么用的、什么时候该往里面写东西。例如：
   - `project.md`：了解项目时先读它；项目架构发生较大调整后更新（架构权威来源）。
   - `learnings.md`：开发经验沉淀；由复盘技能(6)追加。
   - `pending.md`：待确认问题的暂存区；澄清/设计中悬而未决的点先记这里，确认后清理。
   - `log.md`：本上下文目录自身的演进日志（结构/架构级变更）。
   - `rules/`：编码前参考的规则基线；`build.md`/`testing.md` 固化构建/测试命令、`security.md` 固化安全高危点，是下游技能跨语言运行的依据；发现新的团队约定时补充对应规则文件。
   - `requirements/<功能>.md`、`designs/<功能>.md`、`tasks/<功能>.md`：某功能从需求到设计到执行的活文档，同名贯穿。其中 `requirements/<功能>.md` 的 frontmatter `resource` 字段记录该功能的需求来源（PRD、原型设计的链接或路径）——只存引用、不把原件搬进来，原始 PRD/原型留在其权威来源（Figma/蓝湖/Confluence 等）由引用指过去，避免仓库膨胀与双写漂移。
   - `tasks/current.md`：当前在途功能走到哪一步的轻量索引。
   - `archive/<功能>.md`：功能交付后提炼归纳的成品，兼历史需求索引。
3. **Frontmatter 规范** —— 把本技能「Frontmatter 规范」节**完整写入 README**：字段三档（必需 `type`、推荐 `title`/`timestamp`、各 `type` 专属字段）、`type` 取值表、`status` 取值（`draft`/`in-progress`/`reviewing`/`delivered`）、以及"写时按规范填、读时宽容缺失"的消费契约。这是用户要求落到 README 里的核心内容——它让规范随仓库走，而不只活在技能定义里。
4. **交叉链接约定** —— 把本技能「交叉链接约定」节写入 README：活文档开头放 bundle-relative（`/...`）关联指引行、坏链表示"尚未写的知识"且消费方必须容忍。
5. **使用约定 / 工作流说明** —— 说明这套目录如何配合开发流程使用，重点讲清活文档与归档的流转：一个功能从 `requirements/<功能>.md` 起，经 `designs/`、`tasks/` 同名活文档贯穿推进；交付后由 `5-committing-changes` 把活文档**提炼归纳**成 `archive/<功能>.md`（是浓缩成品，不是原样搬运），活文档本身留原处作详细过程记录。

`.ctxloom/README.md` 是本目录里**唯一可带 frontmatter 的 `README.md`**，且仅允许 `okf_version: "0.1"` 一项（用于声明 bundle 遵循的 OKF 版本）；其余目录下的 `README.md` 不带 frontmatter。

写 `README.md` 时用平实的语言，让一个第一次看到这个目录的同事能据此正确使用它，也能照着第 3、4 节正确地给新文档写 frontmatter 与关联链接。
