# 复盘与经验回流

本技能是**任意一次工作收尾后的通用复盘动作**——把这次过程里积累的经验**沉淀回项目**，让下一次不必重踩同样的坑。它的价值在于形成**飞轮**：规则库与经验库随每次开发滚动生长，越用越懂这个项目，而不是每次从零开始。

它有两种入口，地位平等：

- **作为 0–6 流程的收官步骤**——走完 5-committing-changes 提交归档后顺势进入，对刚交付的需求做复盘。这是它最初的设计场景。
- **作为任意工作的独立收尾**——你刚用某个**自定义技能**、或只是手动改了一处东西、查清一个问题，做完后想把经验留下，同样可以直接复盘。**不要求先走过 0–5 的任何阶段**，也不要求这次工作走的是哪条流程。

换句话说，"刚做完一件事，把这次学到的固化成以后都能用的"——只要是这个意图，本技能就适用，无论这件事是怎么做完的。没有这一步，经验只留在你脑子里和对话里，下个需求、下个会话、下个同事都得重新踩。

## 简繁自适应（本技能天然自适应）

本技能**不靠基线档缩放**——它本就按"本次有没有真值得沉淀的经验"自动伸缩：有就记、没有就一句话收尾，不强凑（这正是它的硬性规则之一）。所以无论上游是标准档还是深度档、还是极简档逃生口直接交付，本技能的执行方式都一样：扫一遍本次过程，有料就沉淀、没料就结束。

**硬地板**：再轻也要真扫一遍（步骤 2 的几类信号过一遍），而不是默认"没经验"跳过。判定"无新经验"要基于实际回看，不是图省事。

> **不依赖前序阶段的产物。** 本技能复盘所需的素材来自**本次工作的实际过程**（对话里发生了什么、改了哪些代码、卡在哪、定了什么约定），而不是某个流水线阶段生成的评审报告或归档文件。即使本次工作没走过 0–5、没有 `current.md`、没有归档，也照样能复盘——下面步骤里凡涉及读流水线产物的地方，都是"有则用、没有就从对话与改动本身提炼"，不是前置条件。

> **语言无关原则。** 本技能跨 Java、Vue 等多种项目复用，只描述"如何提炼与沉淀经验"的方法论。提炼出的具体规则内容由项目本身决定，落点是该项目 `.ctxloom/rules/` 与 `learnings.md`，本技能不预设任何语言专属内容。

## 硬性规则

这几条是底线，任何时候都不能违背：

- **规则回写必须经用户确认，不自动塞。** `rules/` 里的规则会约束这个项目未来所有新代码，是"立法"行为。复盘可以**提议**候选规则，但要不要纳入、怎么措辞，必须由用户拍板。绝不自作主张往规则文件里写条目。
- **只沉淀真实发生的经验，不臆造。** 提炼的依据是本次开发**实际**踩的坑、评审/质检**实际**反复打回的点、过程中**确实**临时定下的约定。没有就如实说"本次没有值得沉淀的新经验"，不为了凑数编教训。
- **区分"规则"与"记录"。** 够通用、能指导未来任意新代码的，才提议进 `rules/`；只对本次成立、或偶发的坑，记入 `learnings.md` 即可。一条只对单个文件/单次事件成立的描述不是规则——这与 `0-init-project-context` 对"规则必须通用"的要求一致。
- **不替用户碰 git。** 本技能产出的是对 `.ctxloom/` 下文档的修改（`learnings.md`、规则、`project.md`）。这些改动是否提交、何时提交，交给用户（或下一次 5-committing-changes 顺带），本技能不执行 git 提交。

## 工作流程

> **开始前——检查项目上下文地基。** 本技能要写入 `learnings.md`、`rules/`，可能刷新 `project.md`，这些都在 `.ctxloom/` 下。动手前先看它是否存在：
> - **不存在** → 没有沉淀的落点。提示用户：复盘的经验需要有地方存放，建议先用 `0-init-project-context` 建立上下文目录；或者用户也可以只口头听取复盘结论、不落盘。由用户选。
> - **存在** → 进入步骤 1。若缺 `learnings.md`（老项目未升级），写入前先按 `0-init-project-context`《learnings.md 的内容》的结构创建它（含 `type: Learnings` frontmatter 与表头说明）。
>
> 本技能引用的 `.ctxloom/` 路径以 `0-init-project-context` 的《目录结构》为权威定义，结构调整以其为准。

### 步骤 1：确认复盘对象

先弄清这次复盘的是哪件事。**优先从对话上下文判断**：刚才这个会话里做完的那件工作（实现的功能、修的 bug、用某个自定义技能跑完的任务、查清的问题），通常就是本次复盘对象。

- 能从对话上下文直接确认本次刚做完的事 → 用它，这是最常见的情况。
- 对话里线索不足、或走过了 0–6 流程 → 可**辅助**读 `.ctxloom/tasks/current.md` 与 `.ctxloom/archive/README.md`：刚交付（current.md 中标为已交付、或 archive 索引里新登记）的功能往往就是复盘对象。这两份文件没有也不影响——不是所有工作都会留下它们。
- 有多个候选、或对不上 → 列出来让用户确认复盘哪一个。
- 用户只是泛泛想复盘、没有明确对应的事 → 也可以，按用户描述的范围来。

用一句话把"这次复盘的是哪件事、围绕它聊聊有什么经验值得留下"跟用户对齐，然后进入步骤 2。

### 步骤 2：提炼本次经验

回看本次开发的全过程，从这几个来源挖掘值得沉淀的经验（**有则提炼，没有就少提，不硬凑**）：

- **反复出现的问题** —— 编码中同一类错误犯了不止一次（如总是漏某种空值判断、某类接口总是忘了加校验）。
- **评审/质检反复打回的点** —— 过程中被反复指出、纠正的问题（若走过流水线，就是 1/2 阶段评审、5 阶段质量检查里标为"阻断/重要"且**多次**出现的点；若没走流水线，就是对话里用户反复打回、或你自己反复返工的地方）。这些是项目最该立规矩的地方。
- **临时定下的约定** —— 过程中和用户商量后定的做法（如"这类统计的日期边界统一这么算""这个目录下的组件统一这么组织"），口头定了但还没进规则的。
- **踩的坑与解法** —— 卡了较久、最后找到原因的问题（环境、依赖、框架行为、项目特有的陷阱）。
- **架构认知的变化** —— 这次开发是否让你对项目架构有了新理解，或本次改动本身改变了架构(新增了模块、调整了分层)。

把挖到的点整理成一个简短清单，**逐条标注它的归宿倾向**：是够通用该进 `rules/`（哪个规则文件），还是只记入 `learnings.md`，还是涉及架构、该更新 `project.md`。

把这份清单先**摆给用户看**，对齐"哪些值得沉淀、各自怎么归类"，再进入落盘。没有任何值得沉淀的，如实告诉用户"本次没有新经验值得沉淀"，结束即可。

### 步骤 3：候选规则回写（经用户确认）

对清单里**倾向进 `rules/`** 的条目，逐条把它写成规范的候选规则提给用户：

- **措辞成通用规则**：描述"在这个项目里以后应当怎么做"，能直接套用到未来的新代码上（做一次自检：明天新增一个类/组件，这条能指导它吗？）。参照 `0-init-project-context` 的正反例标准。
- **指明落点**：该进 `code-style.md` / `coding.md` / `build.md` / `testing.md` / `security.md` / `other.md` 的哪一个、追加在哪部分。追加规则时只动正文、保留该规则文件原有的 `type: Rule` frontmatter 不被破坏（若老项目该文件还没 frontmatter，可顺手补上，见 `0-init-project-context`《Frontmatter 规范》）。
- **交用户拍板**：用户确认纳入、并认可措辞后，才追加写入对应规则文件。用户否决或要改措辞的，照用户的来。

逐条处理完，把"哪些规则已写入、写进了哪个文件"汇总告知用户。

### 步骤 4：经验库追加

把清单里**该记入经验库**的条目（包括没被采纳为规则、但值得记的坑），按 `learnings.md` 的结构追加（追加到表头之后、保留文件顶部的 `type: Learnings` frontmatter）：

```markdown
## <日期> <功能名>
- **坑**：<问题现象>
- **根因**：<为什么会这样>
- **解法/约定**：<以后怎么避免>
- **是否已入规则**：是(→ coding.md 第 X 条) / 否(仅记录)
```

已经在步骤 3 提炼为规则的条目，在 `learnings.md` 里也留一条记录、并在"是否已入规则"标明去向，让经验库成为完整的过程档案；纯记录类的标"否"。

### 步骤 5：刷新架构认知（按需）

若步骤 2 发现本次开发改变了对架构的理解、或改动本身调整了架构（新增模块、改了分层、引入新的外部依赖），提示用户更新 `.ctxloom/project.md` 对应章节，并在用户认可后改写（只改正文、保留其 `type: Project` frontmatter）。没有架构层面的变化则跳过。

### 步骤 6：汇总复盘结果

给用户一份简短的复盘小结：

```markdown
## 复盘小结：<需求标题>

### 沉淀为规则
- （写入了哪几条规则、各进了哪个规则文件；无则写"本次无新增规则"）

### 记入经验库
- （追加了哪几条经验；无则写"本次无新增经验记录"）

### 架构认知更新
- （是否更新了 project.md、更新了什么；无则写"无"）

### 结论
（一句话：本次复盘沉淀了什么，对后续开发的帮助。）
```

提醒用户：这些都是对 `.ctxloom/` 下文档的改动，可在合适的时候连同代码一起提交（本技能不替用户操作 git）。

记住硬性规则：规则回写必须经用户确认、只沉淀真实经验、区分规则与记录、不替用户碰 git。
