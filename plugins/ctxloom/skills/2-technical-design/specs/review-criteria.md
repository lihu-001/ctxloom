# 技术设计评审维度与定级标准

## 检查维度

评审智能体按下面的维度审视设计，把发现的问题列成清单并逐条定级。**不打数字分**——逐条列出真实问题、按影响定级，更可靠也更好复核。评审的价值在于诚实：发现多少问题就列多少，**绝不为了让流程赶紧过去而漏报问题、或降级**。

| 维度 | 考察什么 |
| --- | --- |
| 需求覆盖 | 设计是否完整覆盖需求、有无遗漏，有无超出需求的多余设计 |
| 过度设计（重点） | 有无引入需求不需要的抽象 / 扩展点 / 配置化 / 设计模式 |
| 项目规则遵循 | 分层、命名、事务、依赖注入、参数校验、兼容性等约定是否遵循；偏离是否已显式说明并有正当理由 |
| 现有代码一致性 | 设计提到的类/接口/数据表/复用判断是否与现状一致，有无"以为能复用其实不存在""衔接不上" |
| 可落地性 | 关键类/接口/表结构/接口契约是否齐全，开发者能否照着实现 |
| 安全性 | STRIDE 式风险(越权/注入/泄露/篡改/无审计)，按项目规则的安全高危点核对 |

## 定级标准

| 级别 | 含义 |
| --- | --- |
| 阻断 | 设计无法落地 / 与需求或项目现状根本冲突 / 遗漏核心需求 / 存在可被利用的安全漏洞 |
| 重要 | 不影响整体可行，但会导致返工：规则偏离未说明、明显过度设计、复用判断错误、关键契约缺失、需求点遗漏 |
| 建议 | 可优化项、风格不一致、表述不清 |

## 特殊关注点

评审时特别注意：

1. **重点防过度设计**——在满足需求的前提下以简单、易实现为最高准则，任何超出当前需求的抽象/扩展点/配置化/设计模式定为**重要**级
2. **STRIDE 式安全检查**——越权、注入、泄露、篡改、无审计，具体高危模式按 `.ctxloom/rules/security.md`

## 门禁处理

拿到问题清单后（**无数字分，看有没有阻断/重要**）：

- **无阻断、无重要（仅建议或无问题）**：可以定稿。仍要逐条过"建议"，能顺手改的改掉，不改的在记录里说明原因。
- **有阻断或重要**：按评审意见**修订设计文档主体**，优先解决阻断、再重要，然后**重新派发评审智能体**，直到清单里再无阻断/重要。
- 自动"修订→重审"最多进行 **2 轮**。若 2 轮后仍消不掉阻断/重要，不要硬改，把卡住的问题摆给用户。
- 若评审意见与用户已拍板的决策冲突，不要擅自推翻——把冲突点摆给用户，由用户定夺。
