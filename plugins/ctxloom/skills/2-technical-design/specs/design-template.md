# 技术设计文档模板

## 标准结构

技术设计文档严格使用下面的模板结构：

```markdown
# <需求标题> 技术设计文档

## 1. 概述
（一段话说明这次设计要实现什么需求、对应的需求分析文档是哪份、本设计的整体思路。）

## 2. 方案概述与选型
（整体技术方案的设计思路。如果有多个候选方案，说明最终选了哪个、为什么——把关键取舍沉淀下来。说明本方案对现有架构的影响：复用了什么、新增了什么、改动了什么。）

## 3. 类 / 接口 / 分层设计
（涉及的分层与各层职责。新增或改动的关键类、接口及其职责，关键方法的签名与作用。说明它们如何与现有调用链衔接。可用列表或简单的结构描述，必要时给出关键方法签名表达设计意图，但不写完整实现。）

## 4. 数据库设计
（新增或变更的表结构：表名、关键字段及类型、索引、与现有表的关联关系。说明字段设计和索引选择背后的考量。若不涉及数据库改动，写明"本需求不涉及数据库变更"。）

## 5. 对外接口与流程
（新增或变更的接口定义：路径、方法、入参、出参、错误码/异常约定。关键业务流程的步骤编排，复杂流程可用文字描述时序或数据流，说明事务边界、异常处理与并发考量。）

## 6. 风险与待确认项
（设计中存在的风险、与现有架构的潜在冲突、依赖的前置条件、尚未完全确定需要后续跟进的点。**以及对项目规则的遵循与偏离说明**：本设计是否完全遵循 `.ctxloom/rules/` 下的约定；若有偏离，逐条写明偏离了哪条规则、原因与取舍。完全遵循则写明"本设计遵循项目现有规则，无偏离"。没有其他风险则相应写"暂无"。）

## 7. 设计评审记录
（由步骤 6 的独立评审追加，包含评审结果、轮次、问题清单与处理。）
```

## 使用说明

- 模板中不适用于本次需求的章节，保留标题并写明"本需求不涉及"，不要直接删掉
- 保持文档结构统一，便于团队阅读和对照
- 第 7 节由独立评审自动追加，初稿不包含此节
