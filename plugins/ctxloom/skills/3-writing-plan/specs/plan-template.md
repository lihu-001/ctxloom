# 实现计划文档模板

## 标准结构

实现计划文档严格使用下面的模板结构：

```markdown
# <需求标题> 实现计划

## 1. 概述
（一段话说明本计划要落地的是哪份技术设计、整体改动范围、计划共分几步、大致的实现主线。）

## 2. 前置条件
（开始编码前需要就绪的东西：依赖的设计文档、需要的环境/数据/权限、依赖的其他改动是否已合入。没有则写"无"。）

## 3. 实现步骤
（按执行顺序逐步列出。每一步严格用下面的小节格式，步骤之间用序号区分，被依赖的步骤排在前面。）

### 步骤 1：<这一步的简短标题>
- **目标**：这一步要达成什么、做完后项目处于什么状态。
- **改动文件 / 类**：列出新增或修改的具体文件、类、方法（精确到真实路径与名称；新增的写明放在哪个包/目录）。
- **实现要点**：关键逻辑说明，必要时用方法签名或伪代码表达意图，但不写完整实现代码。
- **依赖**：依赖哪些前序步骤（如"依赖步骤 1 的数据结构"）；无则写"无"。
- **测试策略**：test-first / characterization / 可执行验证（三选一）。选"可执行验证"要附一句为什么不单测。
- **验证命令**：这一步做完用什么命令确认没问题——具体命令取自 `.ctxloom/rules/build.md`/`testing.md`（如编译命令、跑某个测试用例的命令、调某个接口看返回）。给出可执行的具体手段，不写死非本项目的命令。

### 步骤 2：……
（同上格式，依此类推。）

## 4. 整体验证
（所有步骤完成后的端到端验证：完整编译、回归测试、关键业务流程手动验证的要点，命令取自 `.ctxloom/rules/build.md`/`testing.md`。说明怎样算"这个需求做完了"。）

## 5. 风险与回滚
（实现过程中的风险点、容易出错的步骤、与现有功能可能的冲突；涉及数据变更或线上影响时，说明回滚思路。没有则写"暂无"。）

## 6. 待确认项
（计划中尚未完全确定、需要在编码中进一步明确或与他人对齐的点。没有则写"暂无"。）
```

## 测试策略三档

| 改动类型 | 测试策略 | 执行时的做法 |
|---|---|---|
| 新增纯业务逻辑（算法、计算、转换、校验等可独立验证的逻辑） | **test-first**（默认） | 先写一个会失败的测试、确认它失败，再写实现让它通过 |
| 改动既有逻辑 | **characterization test** | 先补一个锁定当前行为的测试，再改；改完测试该红就红，确认是预期内的行为变化 |
| 难单测（外部集成、UI/前端交互、批处理/调度、重 IO/中间件） | **可执行验证**（不硬上单测） | 写明具体的手动/集成验证命令与预期结果（起服务调接口看返回、跑 Job 核对落库等） |

默认对业务逻辑步骤选 test-first；选"可执行验证"档的，要在步骤里**写明为什么不能/不值得单测**。

## 使用说明

- 模板中不适用于本次需求的章节，保留标题并写明"本需求不涉及"或"无"，不要直接删掉
- 保持文档结构统一，便于团队阅读和对照
- 验证命令必须读取项目规则（`.ctxloom/rules/build.md` 和 `testing.md`），保持语言无关
