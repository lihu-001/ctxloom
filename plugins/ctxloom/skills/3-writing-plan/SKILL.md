# 编写计划

本技能承接技术设计的产出，帮助你把一份"怎么做"的技术设计，拆解成一份"分几步做、每步做什么、怎么验证"的实现计划。它的价值在于**把蓝图变成可执行的施工序列**——在动手写代码前，先把改动拆成有序、粒度合理、彼此衔接的步骤，让每一步都能独立完成并验证，避免一头扎进去写到一半才发现顺序错了、漏了依赖、或某一大块改动无从下手、难以回头。

## 硬性规则

这几条是本技能的底线，任何时候都不能违背：

- **计划必须基于技术设计文档与项目实际代码，不臆造。** 计划的依据有两个来源：一是技术设计文档（方案选型、分层、表结构、接口契约已经定下来的东西），二是项目里真实存在的代码（要改的文件、类、方法此刻长什么样）。两者都要先核实清楚。设计文档里提到的类、接口、表，要到项目里确认它们真实存在、签名一致；不要凭设计文档的描述就臆想代码现状。
- **拆解计划，不替代设计，更不替代编码。** 关键的技术决策（用什么方案、分几层、什么表结构）已经在技术设计阶段拍板了，本技能不重新做设计；如果在拆解中发现设计有遗漏或硬伤，要指出来并和用户确认，而不是自作主张改设计。同时，你的产出是**施工序列**——每步改哪些文件、做什么、验证什么，是**计划**不是**成品**。可以写关键的方法签名、伪代码、改动要点来表达每步意图，但不要把完整实现代码写进计划文档。
- **每个步骤都必须可独立完成、可验证，并指明测试策略。** 一份好计划的核心，是把大改动切成一连串"做完一步、能确认这步没问题、再走下一步"的小步。每个步骤都要给出**测试策略**与**完成后如何验证**（能编译、相关测试通过、某个接口能跑通等）。无法验证的步骤，意味着它要么太大该拆分，要么定义不清。**测试策略分三档**（详见步骤 3）：①新增纯业务逻辑→默认 **test-first**（先写失败测试再实现）；②改既有逻辑→**characterization test**（先补锁定现状的测试再改）；③难单测的（集成/UI/批处理/重 IO）→不硬上，但必须写明可执行的手动验证命令与预期。产出业务逻辑的步骤默认走 test-first；选其他档要在步骤里写明为什么不能单测。
- **验证命令语言无关、读项目规则。** 计划里每步的验证命令（怎么编译、怎么跑某个测试）不写死某语言的命令，而是采用 `.context/rules/build.md`（构建命令）与 `.context/rules/testing.md`（测试方式）里固化的本项目构建与测试方式。这样同一套计划技能在 Java、Vue 等项目都适用。
- **步骤粒度要适中，顺序要尊重依赖。** 步骤不要大到一步包揽半个功能，也不要碎到每改一行算一步。一个合理的步骤通常对应"一个内聚的改动单元"（如新建一个实体类、加一个 Service 方法、改造一处调用链）。步骤之间有先后依赖的，顺序必须正确——被依赖的先做（如先建表、再写 Mapper、再写 Service、最后接 Controller），让计划顺着做下来始终处于可编译、可验证的状态。
- **工作流程必须严格按步骤执行。** 下面的步骤有先后依赖，每一步都要等用户确认后才能进入下一步。不要跳步，不要把多步合并成一次输出。这个约束是为了让用户在计划落盘前有机会纠偏——计划一旦跑偏，照着它编码的返工成本最高。

## 工作流程

> **开始前——检查项目上下文地基（找不到不阻塞）。** 本技能依赖上游产物：技术设计文档、以及 `.context/` 下的规则与背景。但**每步可独立运行**，缺上游不死等。动手前先看 `.context/` 是否存在：
> - **存在** → 进入步骤 1。落盘时（步骤 5）若 `.context/tasks/` 缺失，写入前先 `mkdir -p` 创建。
> - **不存在** → 优先提示用户先用 `0-init-project-context` 建立上下文地基、并完成上游的需求分析与技术设计，再回到本技能（推荐）。但用户若想直接拆计划，允许**降级**：请用户直接提供技术设计/方案描述，据此拆解实现步骤；缺 `rules/build.md`、`rules/testing.md` 时，步骤的验证命令改为向用户确认本项目怎么构建/测试，并在计划"前置条件"里注明"本次缺项目规则基线，验证命令以用户确认为准"。
>
> 本技能引用的 `.context/` 路径以 `0-init-project-context` 的《目录结构》为权威定义，结构调整以其为准。

### 步骤 1：定位技术设计文档，确认计划目标

实现计划的输入是技术设计文档。**先读 `.context/tasks/current.md`**：若其中记录的当前在途功能已完成「技术设计」、待做「实现计划」，通常就是本次对象，据此定位它的设计活文档 `.context/designs/<功能>.md`（并核对该文件确实存在）。`current.md` 不存在或与实际文档对不上时，再到 `.context/designs/` 下查找：

- **找到唯一一份** → 读取它，向用户确认"我们这次是要给《XXX》这个设计制定实现计划，对吗？"
- **找到多份** → 列出文件名让用户选择是哪一个。
- **一份也没有，或目录不存在** → 询问用户：是否已经做过技术设计？如果有文档请提供路径；如果确实没有，提醒用户实现计划最好建立在技术设计之上，并询问是直接提供设计方案、还是先做技术设计。

读完设计文档后，用 **2-3 句话** 把"这次要落地的核心改动和范围"复述给用户确认。**用户确认后才能进入步骤 2。**

这一步的意义是：确保计划拆解的对象是正确、最新的那份设计，别拿着旧版或不相干的设计去排步骤。

### 步骤 2：核实设计、摸清改动点

在拆步骤之前，先把设计落到真实代码上。这一步不输出给用户，是你自己做功课，目的是让每个步骤都精确指向真实存在的文件与类，而不是悬空的设想。重点核实：

- **设计涉及的代码现状** —— 优先用 CodeGraph 定位设计文档里提到的每一个要新增或改动的模块、类、接口、方法、数据表（`codegraph_search`/`codegraph_node` 确认现有的真实存在、签名/结构与设计描述一致），未启用时退回 grep/读文件；新增的，确认应该放在哪个包/目录下、参照哪个同类文件的写法。
- **改动的波及面** —— 用 `codegraph_callers`/`codegraph_impact` 查要改的方法/接口都有谁在调用、改动会牵连到哪些设计文档没提到的地方（未启用时用 grep 查引用兜底）。把这些牵连点找出来，它们往往是计划里容易遗漏的步骤。
- **项目的验证手段** —— 读 `.context/rules/build.md`（构建命令）与 `.context/rules/testing.md`（测试框架与命令）里固化的本项目方式（缺失或降级模式下向用户确认），明确每步"验证"具体能用什么手段（编译、单测、某个入口手动跑通），以及单测怎么跑单个用例。**不预设某语言的命令**——具体命令以项目规则记录的为准。

摸得越清楚，拆出来的步骤越贴合真实代码，"改动文件""测试策略""验证命令"才填得准。

### 步骤 3：拆解并对齐实现步骤（含测试策略）

基于设计和核实到的代码现状，把整体改动拆成一连串有序、可验证的实现步骤。拆解时，多数情况下你能依据设计直接拍板顺序与粒度；但若遇到**设计没覆盖、或有多种合理拆法/排法**的地方，**逐个**向用户确认，不要自己闷头定：

- **一次只对齐一个点。** 比如"这部分是先改数据层还是先搭接口骨架""这个大改动切成 2 步还是 3 步更好""数据迁移放在哪一步"。每个点附上你的分析和倾向方案，让用户聚焦决策。
- **用户在当前点上拍板后，才能问下一个。** 没有分歧、设计已经讲清楚的部分，不必逐一确认，直接纳入计划即可。

**给每一步定测试策略（语言无关的分层 TDD）。** 拆步骤的同时，为每步选一档测试策略——这决定了步骤 5 模板里"测试策略"字段怎么填、以及执行阶段(4)怎么跑：

| 改动类型 | 测试策略 | 执行时的做法 |
|---|---|---|
| 新增纯业务逻辑（算法、计算、转换、校验等可独立验证的逻辑） | **test-first**（默认） | 先写一个会失败的测试、确认它失败，再写实现让它通过 |
| 改动既有逻辑 | **characterization test** | 先补一个锁定当前行为的测试，再改；改完测试该红就红，确认是预期内的行为变化 |
| 难单测（外部集成、UI/前端交互、批处理/调度、重 IO/中间件） | **可执行验证**（不硬上单测） | 写明具体的手动/集成验证命令与预期结果（起服务调接口看返回、跑 Job 核对落库等） |

默认对业务逻辑步骤选 test-first；选"可执行验证"档的，要在步骤里**写明为什么不能/不值得单测**。这样代码质量从计划阶段就被钉住，而不是事后补。

拆解时心里装着这几条，步骤自然就顺了：

- **按依赖定顺序** —— 底层先于上层（数据层 → 领域/服务层 → 接口层）、被依赖的先于依赖方、数据准备先于使用。（具体分层名称随项目技术栈，参照 `.context/project.md`。）
- **每步自成闭环** —— 一步做完，项目应处于可编译、可验证的状态，而不是半截子改动让代码编译不过。
- **粒度对应一个内聚改动** —— 不贪大、不琐碎，一步大约对应一个开发者能连续完成并自检的工作单元。
- **别漏了配套改动** —— 配置、枚举、常量、数据脚本、测试、对既有调用方的适配，这些容易在只盯着主流程时被落下。

把拆好的步骤列表（含每步的目标、改动文件、测试策略、验证命令）整理出来，进入步骤 4。

**边拆解边落盘（防上下文丢失）。** 拆解与对齐可能要多轮，这份逐渐成形的步骤列表若只留在对话里，一旦上下文被压缩或会话中断就可能丢失。因此每对齐一处拆分/顺序决策、或每完善一批步骤，就把当前的步骤列表（含每步目标、改动文件、测试策略、验证命令）追加/更新写入该功能的任务活文档 `.context/tasks/<功能>.md`（与需求/设计活文档同名，目录缺失先 `mkdir -p`）。**首次创建这份活文档时，按 `0-init-project-context`《Frontmatter 规范》写好头部**（`type: Task`、`feature: <功能>`、`status: in-progress`、`branch:` 此时可留空或占位待执行阶段回填、`title`/`timestamp`），并在正文开头加一行关联指引（`> 关联：需求 [/requirements/<功能>.md] ｜ 设计 [/designs/<功能>.md] ｜ 执行 [/tasks/<功能>.md] ｜ 归档 [/archive/<功能>.md]`，见《交叉链接约定》）。活文档始终反映当前事实、不带日期戳、也不再单开草稿文件。如此决策有外部留存：会话中断可从活文档恢复已对齐内容，步骤 4 的汇总与步骤 5 的成文也以它为据，而非仅凭对话记忆。

### 步骤 4：汇总确认

把完整的实现步骤清单 **统一交给用户确认审核**——包括步骤的划分、顺序、每步改动范围与验证方式。

目的是让用户在落盘成正式文档前，完整回看一遍施工序列，确认拆分合理、顺序无误、没有遗漏。**用户审核通过后才能进入步骤 5。**

### 步骤 5：输出实现计划文档

用户审核通过后，把步骤整合成一份结构化的实现计划，输出成 Markdown 文件。

- 输出文件：`.context/tasks/<功能>.md`（写入前确保 `.context/tasks/` 存在，缺失则先 `mkdir -p` 创建）
- 文件命名：与对应的需求/设计活文档**同名**（如 `coupon.md`），方便顺着同名文件看一个功能从需求到执行的全貌。这是该功能的任务**活文档**——执行阶段(4)会在同一文件里回写每步进度，不带日期戳、不另存历史快照。
- frontmatter 与关联链接：确保文件头部按《Frontmatter 规范》带有 `type: Task`、`feature`、`status: in-progress`、`branch`（执行阶段回填，此处可占位）、`title`、`timestamp`；正文开头保留关联指引行。

文档要让任何开发者照着一步步做下来就能完成实现，而不需要回看对话或反复揣摩设计文档。**严格使用下面的模板结构**：

```markdown
# <需求标题> 实现计划

## 1. 概述
（一段话说明本计划要落地的是哪份技术设计、整体改动范围、计划共分几步、大致的实现主线。）

## 2. 前置条件
（开始编码前需要就绪的东西：依赖的设计文档、需要的环境/数据/权限、依赖的其他改动是否已合入。没有则写"无"。）

## 3. 实现步骤
（按执行顺序逐步列出。每一步严格用下面的小节格式，步骤之间用序号区分，被依赖的步骤排在前面。）

### 步骤 1：<这一步的简短标题>
- **目标**：这一步要达成什么、做完后项目处于什么状态。
- **改动文件 / 类**：列出新增或修改的具体文件、类、方法（精确到真实路径与名称；新增的写明放在哪个包/目录）。
- **实现要点**：关键逻辑说明，必要时用方法签名或伪代码表达意图，但不写完整实现代码。
- **依赖**：依赖哪些前序步骤（如"依赖步骤 1 的数据结构"）；无则写"无"。
- **测试策略**：test-first / characterization / 可执行验证（三选一，见步骤 3）。选"可执行验证"要附一句为什么不单测。
- **验证命令**：这一步做完用什么命令确认没问题——具体命令取自 `.context/rules/build.md`/`testing.md`（如编译命令、跑某个测试用例的命令、调某个接口看返回）。给出可执行的具体手段，不写死非本项目的命令。

### 步骤 2：……
（同上格式，依此类推。）

## 4. 整体验证
（所有步骤完成后的端到端验证：完整编译、回归测试、关键业务流程手动验证的要点，命令取自 `.context/rules/build.md`/`testing.md`。说明怎样算"这个需求做完了"。）

## 5. 风险与回滚
（实现过程中的风险点、容易出错的步骤、与现有功能可能的冲突；涉及数据变更或线上影响时，说明回滚思路。没有则写"暂无"。）

## 6. 待确认项
（计划中尚未完全确定、需要在编码中进一步明确或与他人对齐的点。没有则写"暂无"。）
```

模板中不适用于本次需求的章节，保留标题并写明"本需求不涉及"或"无"，不要直接删掉——保持文档结构统一，便于团队阅读和对照。

计划落盘后，到 `.context/tasks/current.md` 把本功能的当前阶段更新为「实现计划已完成、下一步执行」、更新日期（`current.md` 缺失则跳过并提示用户补全）。

记住硬性规则：写的是分步骤、可验证的施工序列，不是成品代码；每一步都要能独立完成并自检、标明测试策略、顺序要尊重依赖；验证命令读项目规则、语言无关。
