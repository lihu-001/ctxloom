# 编写计划

本技能承接技术设计的产出，帮助你把一份"怎么做"的技术设计，拆解成一份"分几步做、每步做什么、怎么验证"的实现计划。它的价值在于**把蓝图变成可执行的施工序列**——在动手写代码前，先把改动拆成有序、粒度合理、彼此衔接的步骤，让每一步都能独立完成并验证。

## 简繁自适应（按基线档缩放）

本技能支持按任务复杂度缩放深浅。进入时**先读任务活文档 `.ctxloom/tasks/<功能>.md` 的基线档**（`grade: standard|deep`，由 dev 定档写入）：

- **标准档**：改动面不大、步骤心里有数时，不铺开完整的计划文档模板——在任务活文档里列一份**简要步骤清单**（每步：目标 / 改动文件 / 测试策略 / 验证命令）作为执行依据即可。
- **深度档**：完整执行——按 `specs/plan-template.md` 的模板产出结构化的实现计划文档，逐步拆解、对齐顺序与粒度。

**无基线档时**（单独触发、或 dev 未写入）：默认按深度档走；若判断明显偏轻，可降到标准档，但**降档要给反向证据**并一句话告知用户。

**单向放行**：拆解中发现改动比预期复杂（步骤相互纠缠、依赖比想象深）→ 自动加码到更深、铺开完整计划；想做得比基线档更轻 → 停下来向用户说明并确认。

**硬地板（标准档也不能省）**：每个步骤必须**可独立完成、可验证、标明测试策略**——这是计划的立身之本，轻档只缩文档形式（简清单 vs 完整模板），不缩"每步可验证"这条。

## 硬性规则

这几条是本技能的底线，任何时候都不能违背：

- **计划必须基于技术设计文档与项目实际代码，不臆造。** 设计文档里提到的类、接口、表，要到项目里确认它们真实存在、签名一致。
- **拆解计划，不替代设计，更不替代编码。** 关键的技术决策已经在技术设计阶段拍板了，本技能不重新做设计。你的产出是**施工序列**——每步改哪些文件、做什么、验证什么，是**计划**不是**成品**。
- **每个步骤都必须可独立完成、可验证，并指明测试策略。** 一份好计划的核心，是把大改动切成一连串"做完一步、能确认这步没问题、再走下一步"的小步。每个步骤都要给出**测试策略**与**完成后如何验证**。详见 `specs/plan-template.md` 的测试策略三档。
- **验证命令语言无关、读项目规则。** 计划里每步的验证命令（怎么编译、怎么跑某个测试）不写死某语言的命令，而是采用 `.ctxloom/rules/build.md`（构建命令）与 `.ctxloom/rules/testing.md`（测试方式）里固化的本项目构建与测试方式。
- **步骤粒度要适中，顺序要尊重依赖。** 一个合理的步骤通常对应"一个内聚的改动单元"。步骤之间有先后依赖的，顺序必须正确——被依赖的先做。
- **工作流程必须严格按步骤执行。** 下面的步骤有先后依赖，每一步都要等用户确认后才能进入下一步。

## 工作流程

> **开始前——检查项目上下文地基（找不到不阻塞）。** 本技能依赖上游产物：技术设计文档、以及 `.ctxloom/` 下的规则与背景。但**每步可独立运行**，缺上游不死等。动手前先看 `.ctxloom/` 是否存在：
> - **存在** → 进入步骤 1。
> - **不存在** → 优先提示用户先用 `ctxloom:init` 建立上下文地基、并完成上游的需求分析与技术设计（推荐）。但用户若想直接拆计划，允许**降级**：请用户直接提供技术设计/方案描述，据此拆解实现步骤；缺 `rules/build.md`、`rules/testing.md` 时，步骤的验证命令改为向用户确认本项目怎么构建/测试。
>
> 本技能引用的 `.ctxloom/` 路径以 `init` 技能的《目录结构》为权威定义。

### 步骤 1：定位技术设计文档，确认计划目标

实现计划的输入是技术设计文档。**先读 `.ctxloom/tasks/current.md`**：若其中记录的当前在途功能已完成「技术设计」、待做「实现计划」，通常就是本次对象，据此定位它的设计活文档 `.ctxloom/designs/<功能>.md`。`current.md` 不存在或与实际文档对不上时，再到 `.ctxloom/designs/` 下查找：

- **找到唯一一份** → 读取它，向用户确认"我们这次是要给《XXX》这个设计制定实现计划，对吗？"
- **找到多份** → 列出文件名让用户选择是哪一个。
- **一份也没有，或目录不存在** → 询问用户：是否已经做过技术设计？

读完设计文档后，用 **2-3 句话** 把"这次要落地的核心改动和范围"复述给用户确认。**用户确认后才能进入步骤 2。**

### 步骤 2：核实设计、摸清改动点

在拆步骤之前，先把设计落到真实代码上。这一步不输出给用户，是你自己做功课。重点核实：

- **设计涉及的代码现状** —— **配置了 CodeGraph 时，结构性查询必须用它**定位设计文档里提到的每一个要新增或改动的模块、类、接口、方法、数据表（`codegraph_search`/`codegraph_node` 确认现有的真实存在、签名/结构与设计描述一致），只有 CodeGraph 确实未启用时才退回 grep/读文件。
- **改动的波及面** —— 用 `codegraph_callers`/`codegraph_impact` 查要改的方法/接口都有谁在调用、改动会牵连到哪些设计文档没提到的地方（CodeGraph 可用时必须用它）。
- **项目的验证手段** —— 读 `.ctxloom/rules/build.md`（构建命令）与 `.ctxloom/rules/testing.md`（测试框架与命令）里固化的本项目方式，明确每步"验证"具体能用什么手段。

### 步骤 3：拆解并对齐实现步骤（含测试策略）

基于设计和核实到的代码现状，把整体改动拆成一连串有序、可验证的实现步骤。拆解时，多数情况下你能依据设计直接拍板顺序与粒度；但若遇到**设计没覆盖、或有多种合理拆法/排法**的地方，**逐个**向用户确认：

- **一次只对齐一个点。** 比如"这部分是先改数据层还是先搭接口骨架""这个大改动切成 2 步还是 3 步更好"。
- **用户在当前点上拍板后，才能问下一个。** 没有分歧、设计已经讲清楚的部分，不必逐一确认。

**给每一步定测试策略。** 拆步骤的同时，为每步选一档测试策略——详见 `specs/plan-template.md` 的测试策略三档表格。

拆解时心里装着这几条：

- **按依赖定顺序** —— 底层先于上层、被依赖的先于依赖方、数据准备先于使用。
- **每步自成闭环** —— 一步做完，项目应处于可编译、可验证的状态。
- **粒度对应一个内聚改动** —— 不贪大、不琐碎。
- **别漏了配套改动** —— 配置、枚举、常量、数据脚本、测试、对既有调用方的适配。

**边拆解边落盘（防上下文丢失）。** 每对齐一处拆分/顺序决策、或每完善一批步骤，就把当前的步骤列表追加/更新写入该功能的任务活文档 `.ctxloom/tasks/<功能>.md`（与需求/设计活文档同名，目录缺失先 `mkdir -p`）。**首次创建这份活文档时，按 `init` 技能《Frontmatter 规范》写好头部**（`type: Task`、`feature: <功能>`、`status: in-progress`、`branch:` 此时可留空或占位、`title`/`timestamp`），并在正文开头加一行关联指引。

### 步骤 4：汇总确认

把完整的实现步骤清单 **统一交给用户确认审核**——包括步骤的划分、顺序、每步改动范围与验证方式。

目的是让用户在落盘成正式文档前，完整回看一遍施工序列，确认拆分合理、顺序无误、没有遗漏。**用户审核通过后才能进入步骤 5。**

### 步骤 5：输出实现计划文档

用户审核通过后，把步骤整合成一份结构化的实现计划，输出成 Markdown 文件。

- 输出文件：`.ctxloom/tasks/<功能>.md`（写入前确保目录存在）
- 文件命名：与对应的需求/设计活文档**同名**。这是该功能的任务**活文档**——执行阶段(4)会在同一文件里回写每步进度。
- frontmatter 与关联链接：确保文件头部按《Frontmatter 规范》带有 `type: Task`、`feature`、`status: in-progress`、`branch`（执行阶段回填，此处可占位）、`title`、`timestamp`；正文开头保留关联指引行。

文档要让任何开发者照着一步步做下来就能完成实现。**严格使用 `specs/plan-template.md` 的模板结构**。

计划落盘后，到 `.ctxloom/tasks/current.md` 把本功能的当前阶段更新为「实现计划已完成、下一步执行」（`current.md` 缺失则跳过并提示用户补全）。

告知用户：**实现计划已落盘，进入执行阶段前建议先 `/clear`——对话历史可以安全清零，文档不会丢失，下一阶段从干净上下文开始。**

记住硬性规则：写的是分步骤、可验证的施工序列，不是成品代码；每一步都要能独立完成并自检、标明测试策略、顺序要尊重依赖；验证命令读项目规则、语言无关。
