# Bug 修复

本技能帮助你把一个 bug 从"现象"一路收敛到"修好且不再犯"。它和 0–6 那条新功能流水线是**并行的另一条主线**——新功能的难点在"想清楚要做什么"，bug 的难点在"找到为什么坏"，所以它的主轴是：**用证据定位根因**和**按复杂度分流**。

> **语言无关原则。** 本技能跨 Java、Vue 等多种项目复用，只描述"如何复现、定位、验证、沉淀"的方法论。具体的复现命令、测试命令、构建方式以项目 `.ctxloom/rules/` 为准。

## 硬性规则

- **先坐实，再动手——不凭臆测改代码。** 在能稳定复现、或有日志/异常栈/最小复现等**证据**指向之前，不下"根因"结论、不改代码。
- **根因要有证据支撑，不是"看起来像"。** 写进结论的根因，必须能用复现路径、日志、最小复现或代码追踪坐实。
- **只做针对根因的最小必要改动。** bug 修复的范围是"让这个 bug 不再发生"，不顺手重构、不夹带新功能。
- **以"复现步骤不再复现"为验收。** 修完必须用最初那套复现方式重跑一遍确认问题消失。
- **决策权属于用户。** 是否升级走 2/3/4、几个修复方案选哪个、要不要补测试——你可以给推荐，但拍板的是用户。

## 工作流程

### 前置判断：极简逃生口 vs 标准流程

bug 修复默认走标准流程（坐实复现 → 证据定位根因 → 分流），只有**极琐碎、根因一望即知**的 bug 才走逃生口直接修。

**极简逃生口粗筛（四条全中才走逃生口）：**

1. 根因一目了然——用户已经指出了代码位置，或症状直接暴露了根因
2. 改动预计在 5 行以内，不涉及跨模块
3. 无需方案权衡（只有一种显然的修法）
4. 风险低——改错了也容易回退

**极简逃生口只做三件事：**

a. 用一句话确认根因（"我看到是 X 导致的，改 Y"）  
b. 直接改代码  
c. 告知改了什么，以及改完后要怎么验一下

改完后必须调用 `ctxloom:6-retrospective`。**有一丝拿不准，就不算极简，走标准流程**。

### 步骤 1：确认现象与复现

先把"到底哪里不对"问清楚，并设法**稳定复现**。要素：

- **现象**：错误的表现是什么、期望应该是什么
- **触发条件**：什么输入/操作/环境下出现，是必现还是偶现
- **证据**：报错信息、异常栈、相关日志
- **复现路径**：能在本地或测试环境跑出来的最小步骤

把你对"这个 bug 是什么、怎么复现"的理解用一两句话跟用户对齐。**能复现是后面一切的地基**。

### 步骤 2：定位根因（核心）

从复现出发，逐步把范围收敛到**真正的根因**。调试方法详见 `specs/debugging-guide.md`（综合自 David Agans《Debugging》与 Zeller《Why Programs Fail》）。

核心原则：
- 看实际，别靠脑补（加日志、断点、打印中间值）
- 顺着证据走（从异常栈/日志往上游追）
- 二分缩小范围（**CodeGraph 可用时用 `trace`/`callers`/`impact` 快速厘清调用链和波及面**）
- 一次只改一处
- 先查最基础的假设（环境、配置、版本、缓存）
- 区分症状与根因

产出一句话根因 + 支撑它的证据。**没有证据支撑的猜测不能当根因往下走**。

### 步骤 3：分流（triage）+ 根因复核

根因和改动范围摸清后，决定这个 bug 怎么修。**分流不自己拍——用独立子智能体来裁，顺带复核根因。**

派 `general-purpose` 子智能体，给它：根因描述、复现路径、改动范围预估、项目规则路径。它按 `specs/triage-criteria.md` 的标准给出分流建议（简单/复杂）+ 根因复核意见。

主智能体拿到结果后，把分流建议亮给用户确认：
- **简单 bug（简单修复）**：继续步骤 4
- **复杂 bug（升级修复）**：进入步骤 5

### 步骤 4：简单修复（一步到位）

根因清楚、改动小、风险低的 bug，直接改代码。改完后：
1. 按步骤 1 的复现路径验证不再复现
2. 考虑补一个能覆盖该 bug 的测试或防护
3. 进入步骤 6

### 步骤 5：复杂修复（升级走流程）

根因复杂、改动面大、或有多种修复方案需要权衡的 bug，升级走正式流程：

1. **技术设计**（可选）：如果修复方案有多种选择、需要权衡，调用 `ctxloom:2-technical-design` 做设计
2. **编写计划**：调用 `ctxloom:3-writing-plan` 把修复拆成步骤
3. **执行计划**：调用 `ctxloom:4-executing-plan` 按计划落地
4. **验证不复现**：按步骤 1 的复现路径验证
5. **提交**：调用 `ctxloom:5-committing-changes`

### 步骤 6：沉淀经验

所有修复完成、验证不复现后，调用 `ctxloom:6-retrospective` 做复盘：把这次的根因、解法、教训沉淀进 `.ctxloom/learnings.md` 或项目规则（经用户确认）。
