# Bug 修复

本技能帮助你把一个 bug 从"现象"一路收敛到"修好且不再犯"。它和 0–6 那条新功能流水线是**并行的另一条主线**——新功能的难点在"想清楚要做什么"，bug 的难点在"找到为什么坏"，所以它的主轴是 0–6 里没有的两道工序：**用证据定位根因**和**按复杂度分流**。它的价值在于既不让小 bug 被流程拖重，又不让大 bug 被"随手一改"埋雷。

> **语言无关原则。** 本技能跨 Java、Vue 等多种项目复用，只描述"如何复现、定位、验证、沉淀"的方法论，不预设任何语言/框架专属内容。具体的复现命令、测试命令、构建方式以项目 `.context/rules/`（`testing.md`/`build.md` 等）为准；没有这些规则时就问用户或按项目现状判断。

## 硬性规则

这几条是底线，任何时候都不能违背：

- **先坐实，再动手——不凭臆测改代码。** 在能稳定复现、或有日志/异常栈/最小复现等**证据**指向之前，不下"根因"结论、不改代码。bug 修复最常见的翻车就是"看起来像是这里"于是改了一通，结果根因在别处、还引入新问题。证据不足时，宁可多花一轮把现象问清、把复现跑出来。
- **根因要有证据支撑，不是"看起来像"。** 写进结论的根因，必须能用复现路径、日志、最小复现或代码追踪坐实——能说清"为什么是它导致的、改了它为什么就好"。复杂 bug 在修之前起独立智能体复核根因（见步骤 3），用第三方视角挡住自圆其说的假根因。
- **只做针对根因的最小必要改动。** bug 修复的范围是"让这个 bug 不再发生"，不顺手重构、不夹带新功能、不"反正改到了就一起优化"。一旦改动开始超出根因、变成加能力或重构，那已经是新需求/重构，应当回到 0–6 流程走，而不是混在 bug 修复里。
- **以"复现步骤不再复现"为验收。** 修完必须用最初那套复现方式重跑一遍确认问题消失，而不是"改完应该好了"。同时考虑补一个能覆盖该 bug 的测试或防护，挡住它将来回归。
- **决策权属于用户。** 是否升级走 2/3/4、几个修复方案选哪个、要不要补测试——你可以给推荐，但拍板的是用户。

## 工作流程

> **开始前——检查项目上下文地基（找不到不阻塞）。** 本技能对 `.context/` 的依赖比 0–6 更弱：定位和修复一个 bug 本身不需要地基，只有"沉淀经验/升级走 2/3/4"才会用到。动手前看一眼 `.context/` 是否存在：
> - **存在** → 直接进入步骤 1；最后的沉淀（步骤 6）落到 `.context/learnings.md`。
> - **不存在** → 不阻塞，照常修。只是步骤 6 的经验没有落点——届时提示用户：想把这次教训留下，可先用 `0-init-project-context` 建上下文目录，或只口头听取、不落盘。由用户选。
>
> 本技能引用的 `.context/` 路径以 `0-init-project-context` 的《目录结构》为权威定义，结构调整以其为准。

### 步骤 1：确认现象与复现

先把"到底哪里不对"问清楚，并设法**稳定复现**。要素：

- **现象**：错误的表现是什么、期望应该是什么（"点提交后金额算成 0，应该是订单总额"）。
- **触发条件**：什么输入/操作/环境下出现，是必现还是偶现。
- **证据**：报错信息、异常栈、相关日志、出问题的接口/页面/数据。请用户尽量提供，或你去读日志/代码。
- **复现路径**：能在本地或测试环境跑出来的最小步骤。偶现 bug 若一时复现不了，先记录已知触发条件，靠日志和代码追踪推进。

把你对"这个 bug 是什么、怎么复现"的理解用一两句话跟用户对齐。**能复现是后面一切的地基**——复现不了就先解决复现，别急着猜原因。

### 步骤 2：定位根因（核心）

从复现出发，逐步把范围收敛到**真正的根因**，而不是停在表层症状。下面这套手法综合自 David Agans《Debugging》九大规则与 Zeller《Why Programs Fail》的 Delta Debugging，按需取用：

- **看实际，别靠脑补**（Agans: Quit thinking and look）：加日志、断点、打印中间值，去**看**程序真实的运行状态和数据，而不是在脑子里推演"它应该会这么走"。绝大多数假根因都来自"脑补的执行路径"和现实不符。
- **顺着证据走**：从异常栈/日志的报错点往上游追，定位第一个"事实与预期不符"的地方。
- **二分缩小范围**（Agans: Divide and conquer）：在数据流/调用链上分段验证，看问题在哪一段出现，把可疑面一砍再砍。（结构性的"谁调用谁、改这里会牵连什么"可借助 codegraph 的 trace/callers/impact 快速厘清。）
- **一次只改一处**（Agans: Change one thing at a time）：排查时每次只动一个变量并观察结果；同时改多处，就分不清是哪一处起的作用，反而把水搅浑。
- **先查最基础的假设**（Agans: Check the plug）：环境、配置、版本、缓存、是否连错了库/分支——很多"诡异得不可能"的 bug，根因恰恰在这些被默认"不会错"的地方。
- **区分症状与根因**：别停在第一层症状。"金额为 0"是症状，"汇率字段在某分支没赋值"才是根因。改症状会按下葫芦浮起瓢，改根因才一劳永逸。
- **系统化削减出最小复现**（Zeller: Delta Debugging）：不是随手删，而是把可疑的输入/数据/代码变更**二分式地**一批批剥掉，每剥一次就重跑、看是否仍触发——逐步逼近"再删一点就不复现"的最小场景。这个最小集既精确圈定了根因边界，也是将来现成的回归测试雏形。

产出一句话根因 + 支撑它的证据（"改了它为什么就好"要说得清）。**没有证据支撑的猜测不能当根因往下走**，回到步骤 1 补复现/补日志。

### 步骤 3：分流（triage）+ 根因复核

根据根因的清晰度和改动范围，决定这个 bug 怎么修：

- **简单 bug**（根因明确、改动局部、无明显方案权衡）→ 直接进入步骤 4 修掉，不必展开设计/计划，一气呵成。这是大多数情况，别为它套重流程。
- **复杂 bug**（根因深/跨多模块、有多个修复方案要权衡、改动面大或风险高）→ **升级**：把根因分析作为输入，复用现有流程——`2-technical-design` 定修复方案、`3-writing-plan` 拆步骤、`4-executing-plan` 执行。升级后这些阶段会自带它们的评审与验证，本技能在步骤 5 汇总验证结果。

**根因复核（轻量）**：动手改之前，对根因做一次自检——"证据真的能坐实它吗？还有没有别的同样能解释现象的原因？"。复杂 bug 起一个独立智能体，只给它现象、证据和你的根因结论，让它独立判断根因是否成立、有无更可能的解释。这一步是为了挡住"自己说服自己"的假根因——别从一个臆测的根因推导出一通改动。

### 步骤 4：实施修复

- **简单 bug**：针对根因做最小必要改动。遵守项目 `.context/rules/` 的编码规范；改动只围绕根因，不夹带。
- **复杂 bug**：按升级后 `4-executing-plan` 的方式推进（必要时并行步骤用 worktree 隔离、按测试策略验证、回写进度）。

无论哪种，改动都对照硬性规则自检："这是针对根因的最小改动吗？有没有顺手加了不属于本 bug 的东西？"

### 步骤 5：验证不复现 + 防复发

- **重跑复现**：用步骤 1 那套复现方式再走一遍，确认现象消失。这是验收的硬标准，不能用"逻辑上应该好了"代替。
- **查回归**：确认修复没有破坏相邻功能（受影响的同模块/同调用链路径过一遍；改动牵连面可用 codegraph impact 辅助判断）。
- **防复发**：考虑补一个能覆盖该 bug 的测试或加一道防护（按 `.context/rules/testing.md`），让它将来回归时能被自动挡住。要不要补、补到什么程度，给建议由用户定。

把验证结果如实告诉用户：复现是否已不再出现、跑了哪些验证、是否补了测试。没验证通过就不说"修好了"。

### 步骤 6：沉淀

把这次的教训留下，避免下次重踩。轻量做法：按 `learnings.md` 的结构追加一条（追加到表头之后、保留文件顶部 `type: Learnings` frontmatter；缺该文件则按 `0-init-project-context`《learnings.md 的内容》先建）：

```markdown
## <日期> <bug 简述>
- **坑**：<错误现象 / 怎么触发的>
- **根因**：<真正的原因>
- **解法/约定**：<怎么修的；以后怎么避免>
- **是否已入规则**：是(→ 某规则文件第 X 条) / 否(仅记录)
```

判断升级：如果这个 bug 暴露的是一类**反复出现**的问题、或过程中和用户定下了通用约定（够指导未来所有新代码），那它值得提炼成规则——这时走 `6-retrospective` 做完整复盘，把候选规则经用户确认回写 `rules/`。只对本次成立的偶发坑，记入 `learnings.md` 即可。`.context/` 不存在时，口头给出"这次的教训是什么"即可。

最后给用户一份简短小结：

```markdown
## Bug 修复小结：<bug 简述>

### 根因
（一句话根因 + 关键证据）

### 修复
（改了什么；简单修复 / 升级走了 2-3-4）

### 验证
（复现已消失；跑了哪些验证；是否补了测试/防护）

### 沉淀
（记入 learnings 的一条 / 提炼了哪条规则 / 本次无需沉淀）
```

记住硬性规则：先坐实再动手、根因要有证据、只做最小必要改动、以复现不再出现为验收、决策权属用户。
