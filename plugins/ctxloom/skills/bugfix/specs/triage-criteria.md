# Bug 分流标准

## 简单 vs 复杂的判断标准

### 简单 bug（简单修复）

满足以下**所有**条件：

1. **根因明确**：有充分证据支撑，复核后确认无误
2. **改动小**：预计改动 ≤ 20 行，涉及 ≤ 3 个文件
3. **无方案分歧**：只有一种显然的修法，无需权衡
4. **风险低**：
   - 不涉及核心流程或多模块交互
   - 改动范围清晰、不易牵连其他功能
   - 易于回退
5. **测试覆盖**：该部分已有测试，或容易补充测试

### 复杂 bug（升级修复）

只要有**任一**条件满足：

1. **根因不确定**：证据不足，或有多个可能的根因需要进一步验证
2. **改动大**：预计改动 > 20 行，或涉及 > 3 个文件
3. **有方案分歧**：有多种修复思路，需要权衡利弊
4. **风险高**：
   - 涉及核心流程、数据迁移、安全敏感区域
   - 改动可能牵连多个模块
   - 需要数据库变更或配置变更
5. **测试困难**：该部分缺少测试覆盖，补充测试成本高

## 根因复核检查点

独立子智能体在分流时应同时复核根因，检查：

1. **证据是否充分**：日志、异常栈、最小复现能否坐实根因
2. **逻辑是否自洽**：改了它为什么就好，说得清吗
3. **有无遗漏**：是否还有其他可能的根因未排除
4. **修复方案是否针对根因**：确认不是只改症状

如果复核发现根因证据不足，建议回到步骤 2 补充证据。
