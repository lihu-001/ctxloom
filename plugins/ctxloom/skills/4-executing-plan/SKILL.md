# 执行计划

本技能承接 3-writing-plan 的产出，把一份"分几步做、每步做什么、怎么验证"的实现计划，真正落地成代码。它的价值在于**让计划照着既定的施工序列稳稳跑完**——按步骤执行、每步执行完就地验证、把进度回写到计划文档，最终交付一份可编译、经过端到端验证的实现，以及一份让用户一眼看清"做了什么、验证了什么、还剩什么"的执行报告。

## 简繁自适应（按基线档缩放）

本技能支持按任务复杂度缩放深浅。进入时**先读任务活文档 `.ctxloom/tasks/<功能>.md` 的基线档**（`grade: standard|deep`，由 dev 定档写入）：

- **标准档**：依据可能是上一步的简要步骤清单而非完整计划文档。步骤少、彼此无并行价值时，**串行执行、不开 worktree、不强行分派多子智能体**；执行主体可由主智能体直接承接。
- **深度档**：完整执行——按依赖拓扑分派子智能体（并行步骤用 worktree 隔离）、合并后整体验证、产出完整执行报告。

**无基线档时**（单独触发、或 dev 未写入）：按手上计划的规模自然决定串行/并行；拿不准按更稳的深度档做法走。

**单向放行**：执行中发现改动比预期复杂（牵连面超出计划、出现计划没覆盖的步骤）→ 按硬性规则停下来与用户确认（这本就是停手方向）；想省掉验证、跳过某步 → 绝不允许，验证是硬地板。

**硬地板（标准档也不能省）**：① 每步执行完**必须验证、验证不过不算完成**；② test-first 步骤的**先红后绿留痕**不可省；③ 全部完成后做一次整体编译/回归。轻档只缩"分派与隔离"的机制开销，不缩任何验证。轻 ≠ 不验证。

## 硬性规则

这几条是本技能的底线，任何时候都不能违背：

- **严格照计划执行，不偏离、不擅自加戏。** 执行的依据是实现计划文档里已经定好的步骤、顺序、改动范围与验证方式。不要在计划之外顺手重构、改动无关代码、或自行扩大范围。如果执行中发现计划有遗漏、硬伤、或某步根本做不下去，**停下来向用户说明**并确认怎么处理，而不是自作主张改方案或跳过。
- **尊重步骤间的依赖顺序。** 计划里被依赖的步骤排在前面（如先建数据层、再服务层、最后接口层）。执行时必须顺着依赖走——前序步骤没完成、没验证通过，就不能动依赖它的后续步骤。即使用子智能体并行，也只能并行**彼此无依赖**的步骤。
- **按计划的测试策略走，test-first 要先红后绿留痕。** 每步按计划里写明的测试策略执行：**test-first** 步骤必须先写出会失败的测试、确认它失败（红），再写实现让它通过（绿），并在该步的产出里留下"先红后绿"的痕迹（失败输出 + 通过输出）；**characterization** 步骤先补锁定现状的测试再改；**可执行验证** 步骤按计划写明的命令与预期验证。没留下先红后绿痕迹的 test-first 步骤，不算完成。
- **每步执行完必须验证，验证不过不算完成。** 每个步骤都按计划里写明的验证命令自检。验证不通过，就在这一步把问题解决掉，而不是带着错误进入下一步——越往后拖，问题越难定位、返工越贵。（并行步骤的统一编译/回归验证放在步骤 4 合并后做，见下。）
- **验证命令语言无关、读项目规则。** 具体怎么编译、怎么跑测试，采用 `.ctxloom/rules/build.md`（构建命令）与 `.ctxloom/rules/testing.md`（测试方式）固化的本项目命令，不写死某语言的命令。
- **每步完成后回写计划文档标注进度。** 一步做完且验证通过，就到该功能的任务活文档 `.ctxloom/tasks/<功能>.md` 里把该步标注为已完成。这让执行过程随时可中断、可续接：哪怕中途停了，回头看文档就知道做到哪了，不用翻对话重新捋。
- **不替用户操作 git（worktree 除外）。** 全部完成后只提示用户自行验证结果、自行提交，绝不替用户执行 commit / push。但为并行隔离而创建/合并/清理 worktree（见步骤 3）属于执行机制、不是把成果落库，可以做——它不产生提交、不推远端，成果最终仍合并回用户的工作分支由用户把关。

## 工作流程

> **开始前——检查项目上下文地基（找不到不阻塞）。** 本技能依赖上游产物：实现计划文档（位于 `.ctxloom/tasks/<功能>.md`）。但**每步可独立运行**。动手前先看 `.ctxloom/` 是否存在：
> - **存在** → 进入步骤 1。（本技能回写进度到既有任务活文档，不新建上下文目录。）
> - **不存在** → 优先提示用户先用 `0-init-project-context` 建立地基、并完成需求分析/技术设计/编写计划，再回到本技能（推荐）。但用户若已有计划（哪怕只是口头或在别处写的步骤），允许**降级**：请用户直接提供要执行的步骤清单，据此执行；缺 `rules/build.md`、`rules/testing.md` 时，向用户确认本项目的构建/测试命令后再落地验证。
>
> 本技能引用的 `.ctxloom/` 路径以 `0-init-project-context` 的《目录结构》为权威定义，结构调整以其为准。

### 步骤 1：定位实现计划文档，确认执行目标

执行计划的输入是实现计划文档。**先读 `.ctxloom/tasks/current.md`**：若其中记录的当前在途功能已完成「实现计划」、待做「执行」，通常就是本次对象，据此定位它的任务活文档 `.ctxloom/tasks/<功能>.md`（并核对该文件确实存在）。`current.md` 不存在或与实际文档对不上时，再到 `.ctxloom/tasks/` 下查找：

- **找到唯一一份** → 读取它，向用户确认"我们这次是要执行《XXX》这份实现计划，对吗？"
- **找到多份** → 列出文件名让用户选择是哪一个。
- **一份也没有，或目录不存在** → 询问用户：是否已经做过编写计划？如果有计划文档请提供路径；如果确实没有，提醒用户执行计划要建立在实现计划之上，并建议先用 3-writing-plan 技能编写计划。

读完计划文档后，用 **2-3 句话** 把"这次要执行的计划主线、共几步、产物是什么"复述给用户确认。

**确认执行分支。** 执行计划会动较多代码，应该在专门的分支上做，便于隔离改动、回滚和 review，避免污染当前分支。在复述计划时一并和用户确认分支：

- **用户指定了分支名** → 用用户给的名字。
- **用户没指定** → 你自行起一个合理的分支名（如 `feat/<需求英文简称>`、`feature/<需求标题拼音或英文>`），告知用户你将用这个名字，由用户拍板或改名。

分支名敲定后，在该分支上执行后续步骤，并把分支名记入 `.ctxloom/tasks/current.md` 本功能的「关联分支」、同时回填任务活文档 `.ctxloom/tasks/<功能>.md` frontmatter 的 `branch` 字段（`current.md`/活文档缺失则跳过对应一项）。建分支属于改变仓库状态的动作，要在用户确认计划与分支之后再切，不要在用户还没点头时就擅自 checkout。**用户确认计划与分支后才能进入步骤 2。**

这一步的意义是：确保照着执行的是正确、最新的那份计划，并且改动落在一条干净、可控的分支上，别拿着旧版计划开干、也别把改动直接糊到当前分支上。

### 步骤 2：摸清执行环境与验证手段

正式动手前，先把执行所需的底摸清。这一步不输出给用户，是你自己做功课，目的是让执行和验证都落在真实环境上：

- **核对计划与代码现状是否仍然吻合** —— 计划是之前写的，代码可能已有变化。优先用 CodeGraph 确认计划里要改动的文件、类、方法此刻仍如计划所述存在、签名一致（`codegraph_search`/`codegraph_node` 核对符号与签名、`codegraph_impact` 复核波及面），未启用时退回 grep/读文件。若发现明显出入（计划提到的类已不在、签名变了），这是计划过时的信号，按硬性规则停下来与用户确认。
- **确认构建与测试手段** —— 读 `.ctxloom/rules/build.md`（构建命令）与 `.ctxloom/rules/testing.md`（测试框架与命令）固化的本项目方式（缺失/降级时向用户确认），明确每步验证、以及步骤 4 整体验证具体用什么命令落地（编译、跑某个测试、起服务调接口）。**不写死某语言的命令**——以项目规则记录的为准。这是后面验证的基础。
- **理清步骤的依赖拓扑** —— 把计划的步骤依赖关系理清楚：哪些步骤串行（有先后依赖）、哪些彼此独立可并行。这决定步骤 3 怎么分派、要不要开 worktree。

环境摸清了，执行才不会卡在"不知道怎么验证"或"并行了有依赖的步骤"。

### 步骤 3：按步骤执行计划

按计划的依赖顺序，从头到尾连续执行，中途不必逐步等用户确认——计划已经在 3-writing-plan 阶段对齐审核过，这里要的是稳稳跑完。用**子智能体**承接具体步骤的编码，遵循下面的方式：

- **派活看依赖，并行才开 worktree。**
  - **有先后依赖的步骤**：必须等前序完成并验证通过，再派后续；**串行执行、不开 worktree**（worktree 有创建/合并开销，串行用不上）。绝不把依赖链上的步骤并行出去。
  - **彼此无依赖的步骤**：可分派多个子智能体并行（最多 5 个并发）。**并行时每个子智能体用 `isolation: "worktree"` 在独立 worktree 里改文件**，避免多个子智能体同时写同一工作区造成的文件冲突。
- **worktree 内只写代码，不在 worktree 内编译/装依赖。** 给并行子智能体的指令限定为"按本步要点改/写指定文件、对 test-first 步骤先写失败测试再写实现"。**不要求它在 worktree 内跑构建或装依赖**（前端 worktree 没有 node_modules、后端没有本地依赖缓存，逐个重装既慢又费磁盘）——编译与跑测试统一放到步骤 4 合并回主工作区后做。
- **给子智能体的任务要自包含。** 每个子智能体的指令里写清：这一步的目标、要改动的具体文件/方法、实现要点、测试策略（test-first 要先红后绿）。让它能独立完成，而不是回头问。
- **并行批完成后把改动收拢到主工作区。** 一批并行子智能体都完成后，确认每个隔离 worktree 里的改动都落到了主工作区的对应文件（子智能体的 `isolation: "worktree"` 由 harness 管理，无改动的 worktree 会自动清理；若某个 worktree 的改动没有自动体现到主工作区，就把它的产出对照应用过去）。多个步骤改到同一文件而产生冲突时，就地解决或停下来与用户确认。收拢完成后再在主工作区做这一批的验证。
- **每步（或每批）落地后做两件事**：
  1. **代码质量把关** —— 检查产出代码是否符合项目既有规范与写法（参照 `.ctxloom/rules/`）、有无坏味道（重复、命名混乱、超长方法）、有无明显漏洞（未校验入参、注入风险、资源未释放等——具体高危模式见 `.ctxloom/rules/security.md` 的安全高危点）。发现问题就地修正。
  2. **回写计划文档标注进度** —— 到该功能的任务活文档 `.ctxloom/tasks/<功能>.md` 里把这一步标记为已完成（如在步骤标题后加 `✅ 已完成` 或勾选）。

> **关于"就地验证"与"合并后统一验证"的分工。** test-first 的先红后绿是逻辑层面的自检，子智能体在写代码时就该完成(它能在自己的上下文里跑该测试用例)。而**整体编译、跨步骤的回归**这类需要完整工作区的验证，统一放到步骤 4——因为并行 worktree 各自不具备完整可构建环境。串行步骤若环境就绪，也可以就地编译验证，不必非等到最后。

执行过程中随时把当前进展简要同步给用户（做到第几步、有没有遇到卡点），但不为每一步停下等确认。只有遇到"计划做不下去 / 与现状冲突 / 需要超出计划的改动"这类情况，才停下来请用户定夺。

全部步骤执行并逐个验证通过后，进入步骤 4。

### 步骤 4：合并后整体验证

所有步骤完成、并行批次都已合并回主工作区后，在主工作区按计划文档"整体验证"章节做端到端把关，确认整个需求真的落地了，而不只是每步各自通过。**先合并、先精简、再验证**，让被验证的代码就是最终交付的版本：

- **确认已全部合并回主工作区** —— 步骤 3 里并行用过 worktree 的，先确认改动都已合并、worktree 已清理，主工作区是完整的最终代码。整体验证一律在主工作区做（worktree 不具备完整可构建环境）。
- **代码精简与优化** —— 跑一遍 `/simplify` 对本次执行产出的代码做精简与优化。执行过程中为了一步步推进、就地修问题，代码难免留下重复、冗余、命名或结构上的粗糙处，在整体验证前统一收口；放在验证之前，也保证后面编译、回归跑的就是精简后的最终代码。
- **完整编译** —— 整体构建一次，确保所有改动合在一起能编译/构建通过。**命令取自 `.ctxloom/rules/build.md`**（不写死 `mvn`/`npm` 等）。
- **回归与关键流程** —— 跑计划指明的回归测试（含本次新增的 test-first/characterization 测试，应全绿）；对关键业务流程做手动验证（按项目形态：起服务调通主链路接口、跑 Job 核对落库、前端页面走通交互等）。
- **比对验收标准** —— 对照计划"整体验证"里写的"怎样算这个需求做完了"，逐条核对。

整体验证有不通过的，定位并解决，直到达到计划设定的完成标准。

### 步骤 5：汇总执行报告并交付

执行与验证全部完成后，给用户一份清晰的执行报告，让用户一眼看清结果。报告**用下面的结构**：

```markdown
## 执行报告：<需求标题>

### 执行概览
（一句话说明执行了哪份计划、共几步、整体结果如何。）

### 步骤执行情况
（逐步列出，每步一行：步骤名 + 状态（✅完成 / ⚠️有调整） + 一句话说明做了什么、验证结果。若某步相对计划有调整，写明调整原因。）

### 整体验证结果
（`/simplify` 代码精简是否已执行；精简后的完整编译、回归测试、关键流程验证分别的结果。怎样算"做完了"以及是否达成。）

### 改动清单
（本次执行新增/修改的主要文件，按模块或步骤归类，方便用户 review。）

### 遗留与待确认项
（执行中发现但未处理的问题、相对计划的偏差、需要用户进一步确认或后续跟进的点。没有则写"暂无"。）
```

报告给出后，**提示用户自行验证结果，并提示用户自行用 git 提交**——明确告诉用户你不会替他操作 git（为并行隔离用过的 worktree 不算，那是执行机制且已合并清理），提交与否、提交什么由用户把关。可建议用户接着用 `5-committing-changes` 做收尾提交与归档。

整体验证通过、报告给出后，到 `.ctxloom/tasks/current.md` 把本功能的当前阶段更新为「执行已完成、下一步提交归档」、更新日期（执行进行中可临时标「执行中」；`current.md` 缺失则跳过）。任务活文档 `.ctxloom/tasks/<功能>.md` frontmatter 的 `status` 此时仍保持 `in-progress`（功能要到 `5-committing-changes` 归档时才置 `delivered`）。

记住硬性规则：严格照计划执行、尊重依赖顺序、按测试策略执行(test-first 先红后绿留痕)、并行才开 worktree 且 worktree 内只写代码、合并回主工作区后统一验证、验证命令读项目规则、不替用户碰 git。
